# Plan 0625-v2: 基于 `audit0625_kimi.md` 的 `rtlgen_x` 可用性改进计划

## 1. 背景

`audit0625_kimi.md` 不是一次普通的设计总结，它把我们在用 `rtlgen_x`
完成 `jpeg_decoder/` 真实设计时遇到的“不顺手”之处暴露得很具体。

这次暴露出来的问题有一个共同特点：

1. **不是“工具完全不能做”**
2. **而是“能做，但容易踩语义坑，或者需要知道内部 work-around 才能做稳”**

这说明当前 `rtlgen_x` 的主问题已经不再是“有没有能力”，而是：

1. **语义是否足够稳定**
2. **跨后端是否足够一致**
3. **层次化写法是否足够自然**
4. **报错/诊断是否能把用户直接带到正确写法**
5. **stdlib / 文档 / 模板是否能减少重复试错**

因此，这份计划不以“再堆新功能”为目标，而是围绕这次 JPEG 设计过程里暴露出的
真实 friction，继续把 `rtlgen_x` 收紧成一个更稳、更清楚、更适合 agent 和人工共同使用
的 DSL-first 硬件能力栈。

---

## 2. 从 `audit0625_kimi.md` 提炼出的核心问题

### 2.1 P0: signed arithmetic 语义仍然太容易误用

JPEG IDCT 里最关键的问题，不是“不会写乘法”，而是：

1. Python 仿真、C++ 编译仿真、Verilog emitter 对 signed/unsigned 的默认推断并不完全一致
2. 右移、乘法、比较这些位置，如果没有显式 `.as_sint()`，用户很容易写出在 Python 下正确、
   但在 C++ backend / Verilog 语义上出错的代码
3. 当前框架已经能处理显式 signed intent，但还没有把这种 intent 变成“默认容易写对”的体验

这类问题属于 **高优先级语义坑**，因为它不是简单报错，而是可能产生“仿真通过但语义偏掉”
的隐性 bug。

### 2.2 P0: 层次化模块写法仍然不够稳

JPEG 顶层在做 entropy / dequant / idct 级联时暴露了几件事：

1. parent-owned interconnect wire 传给子模块端口时，ownership/lint 很容易误判
2. flattening 路径对一些合法层次结构仍然敏感
3. 有些写法“理论上应该成立”，但实际上要依赖内部实现细节才稳

这说明当前框架虽然支持层次化设计，但“多子模块 + parent interconnect + lowering/lint”
这一条主路径还没有完全成为可靠 happy path。

### 2.3 P0: `Array` / `Memory` / flattening 组合仍然脆弱

这次 JPEG 设计里，transpose buffer、zig-zag buffer、LUT/ROM 都是刚需。

暴露出的问题包括：

1. submodule 内部 `Array` 读取在 flatten 后容易出错
2. `Memory(init_data=...)` 在某些层次化场景下不够稳
3. author 需要知道“哪些存储对象应该放 top、哪些应该 reset 初始化、哪些要先接到 wire”

这意味着存储语义虽然可用，但还没有达到“用户自然写，框架自然兜住”的程度。

### 2.4 P1: 本地临时信号 / 结构化 authoring intent 仍然不够强

`audit0625_kimi.md` 明确提到：

1. 本地 Python 变量形式的 `Wire(...)` 不会自动进入模块跟踪表
2. 用户必须知道“要挂到 `self.xxx` 上”才能稳定 lower / lint / emit

这类问题会强依赖内部约定，agent 也很容易踩中。

### 2.5 P1: 诊断信息还没有足够“带路”

当前很多问题不是完全没有报错，而是：

1. 报错位置不够贴近用户源码
2. 报错没有直接指出推荐修法
3. 某些问题只有到 flatten / lower / backend 阶段才暴露

对于 agent 和人工协作来说，这类问题会显著拉长 debug 回合数。

### 2.6 P1: 文档与 cookbook 仍不足以覆盖真实 datapath 设计

JPEG 这个例子说明，当前框架最需要的不是再写一层总控，而是：

1. 把 signed fixed-point datapath 的正确写法讲清楚
2. 把 `Array/Memory/LUT` 的推荐写法讲清楚
3. 把层次化互连、父模块 wire、flatten-safe pattern 讲清楚
4. 给出 end-to-end worked example，而不是只给孤立 API

---

## 3. 总体改进原则

基于这次审计，后续改进遵循下面几条原则：

1. **优先修“容易写错但不容易第一时间发现”的问题**
2. **优先把当前已经存在的能力变成稳定 happy path**
3. **优先增加 lint / diagnostics / docs，而不是引入新控制层**
4. **优先保证 Python / C++ / emitted RTL / verify 几条主路径语义一致**
5. **优先让真实 datapath 设计更顺手，而不是补表面功能**

---

## 4. 执行主线

本轮按六条主线推进：

1. **W1: signed / fixed-point 语义硬化**
2. **W2: 层次化连接与 flattening 稳定化**
3. **W3: 存储对象 (`Array` / `Memory` / LUT) 语义收口**
4. **W4: authoring intent / diagnostics / source mapping 增强**
5. **W5: JPEG 类 datapath 的 stdlib 文档与 worked example 补强**
6. **W6: DSL / Verilog 混合建模与协同仿真**

每条主线都要求：

1. 有代码改动
2. 有测试回归
3. 有文档更新
4. 有清晰验收口径

## 5. W1: signed / fixed-point 语义硬化

### 5.1 目标

让 signed arithmetic 不再主要依赖“作者自己记得 everywhere 加 `.as_sint()`”。

### 5.2 具体改动

1. 为下列操作补强 authoring-time lint / lowering-time lint：
   - signed value 参与 `>>` / `>>>`
   - mixed signed/unsigned multiply
   - mixed signed/unsigned compare
   - signed rounding pattern
2. 对容易出错的表达式给出更直接的建议：
   - 指明信号名
   - 指明推荐包裹位置
   - 指明建议 `.as_sint()` / `.as_uint()`
3. 补一组 datapath helper / idiom：
   - arithmetic shift helper
   - fixed-point rounding helper
   - saturating clip helper
   - signed multiply / MAC cookbook
4. 对 Python runtime、C++ backend、reference runtime、Verilog emitter 建立统一回归集：
   - signed multiply
   - arithmetic shift
   - negative intermediate rounding
   - LUT + signed datapath chain

### 5.3 验收

1. JPEG IDCT 类 signed datapath 不再需要靠隐含经验才能写对
2. 新增 lint 能在 lower 前或 emit 前报告最常见 signed trap
3. 同一批 signed regression 在 Python / C++ / emitted RTL 上一致通过

---

## 6. W2: 层次化连接与 flattening 稳定化

### 6.1 目标

把“多子模块 + parent interconnect wire + lowering / lint / emit”做成可靠主路径。

### 6.2 具体改动

1. 系统梳理 parent-owned signal 的 ownership 规则：
   - parent local interconnect
   - child port binding
   - flatten 后信号归属
2. 收紧并文档化层次化设计的合法模式：
   - parent wire connect child ports
   - child output -> parent wire -> sibling input
   - local handshake/control wires
3. 对 linter 的 hierarchical read/write 检查继续做语义化修正：
   - 区分“真实越层访问”与“合法的 parent interconnect”
4. 为 flattening 增加更完整的 regression：
   - two-stage datapath
   - three-stage handshake chain
   - array/memory read through submodule
   - parent FSM + child datapath
5. 如果某些模式仍不准备支持，要明确 fail-fast：
   - 明确报 unsupported pattern
   - 给出推荐改写法

### 6.3 验收

1. JPEG 这种三级子模块串接拓扑能稳定 lower / lint / emit
2. 层次化误报显著减少
3. 对未支持模式，报错足够清晰，不再是内部异常或含糊 lint

---

## 7. W3: 存储对象语义收口

### 7.1 目标

让 `Array` / `Memory` / LUT 在 datapath 设计里可预测、可 flatten、可 emit、可仿真。

### 7.2 具体改动

1. 明确区分几类存储对象的推荐用途：
   - `Array`: 小容量结构化寄存器/缓冲
   - `Memory`: RAM/ROM 风格存储
   - LUT helper: 常量查找表
2. 补齐 flatten / rename / lowering 对存储引用的覆盖：
   - `ArrayRead`
   - `ArrayWrite`
   - memory read/write rename
   - nested expression use
3. 明确初始化策略：
   - `init_data` 何时可靠
   - reset initialization 何时推荐
   - submodule-local ROM/LUT 推荐写法
4. 增加存储使用 lint：
   - untracked local storage objects
   - flatten-unsafe pattern
   - unsupported dynamic indexing pattern
5. 建立一组存储回归模块：
   - transpose buffer
   - zig-zag reorder buffer
   - ROM-backed LUT datapath
   - FIFO-like arrayed register bank

### 7.3 验收

1. JPEG 中的 `tbuf`、zig-zag、cos LUT 这类模式都能稳定闭环
2. 存储初始化边界在文档中写清楚
3. 用户不再需要通过试错猜测“这个存储对象该怎么放才稳”

---

## 8. W4: authoring intent / diagnostics / source mapping 增强

### 8.1 目标

把当前“知道内部约定才能写稳”的部分继续显式化，并把报错变成真正可操作的指导。

### 8.2 具体改动

1. 对未注册本地信号增强检测：
   - `x = Wire(...)` 但未挂到 `self.x`
   - `Array(...)` / `Memory(...)` 的局部临时对象未进入跟踪表
2. 在 emit / lower / verify 入口统一做 authoring intent validation
3. 提升 source mapping：
   - diagnostics 尽量指向 DSL 源位置
   - diagnostics 带 signal/module 名称
   - diagnostics 给出“推荐改法”
4. 新增常见错误专门报错模板：
   - signed shift trap
   - parent-child ownership mismatch
   - local untracked signal
   - flatten-unsafe array access
5. 为 agent 友好输出保留统一 finding schema：
   - severity
   - location
   - object name
   - suggested fix

### 8.3 验收

1. 常见作者错误能在更早阶段被发现
2. 报错对人工和 agent 都更可消费
3. 至少覆盖 JPEG 审计里提到的几类高频坑

---

## 9. W5: datapath-focused docs / stdlib / example 补强

### 9.1 目标

把这次 JPEG 设计过程沉淀成可复用经验，而不是停留在一次性修 bug。

### 9.2 具体改动

1. 在 `rtlgen_x/README.md` 或单独 tutorial 中补充：
   - signed fixed-point datapath recommended patterns
   - `Array/Memory/LUT` usage guide
   - flatten-safe hierarchical wiring guide
   - generated RTL / C++ / Python / iverilog closure flow
2. 把 `jpeg_decoder/` 定位成真实 worked example：
   - 标注哪些地方是当前推荐写法
   - 标注哪些地方是框架边界/注意事项
3. 把从 JPEG 里抽象出的共性对象补进 stdlib/cookbook：
   - transpose buffer idiom
   - coefficient reorder idiom
   - signed LUT MAC idiom
4. 给出“从 DSL 到 emitted RTL 再到 cosim/verify”的标准操作流程

### 9.3 验收

1. 新用户可以按文档复现 JPEG 类 datapath 的主闭环
2. 后续再做类似 DSP/codec/SFU datapath 时，重复踩坑明显减少

## 10. W6: DSL / Verilog 混合建模与协同仿真

### 10.1 目标

让 DSL 可以自然地承载“主体用 DSL 写，局部复用现成 Verilog module”的真实工程形态，
并把这条路径收敛成可 author、可 emit、可 verify、可协同仿真的正式能力。

### 10.2 具体改动

1. 增加 DSL authoring 语法/约定，支持在 DSL 中例化外部 Verilog module：
   - 声明外部 module 端口
   - 显式指定外部 Verilog module 名称
   - 支持 parameter override
   - 支持 parameterized module 的多实例绑定
2. 明确区分两类子模块：
   - 可由 DSL 自身生成 RTL 的普通 DSL submodule
   - 仅在 emitted RTL / cosim 中出现、实现来自外部 Verilog 文件的 external/blackbox module
3. 为 mixed design 建立 emit/closure 规则：
   - `emit_design(...)` 不应错误展开 external Verilog module 的内部壳体
   - mixed design 需要支持附带 external Verilog sources / include dirs / defines
   - diagnostics 要能指出“该模块需要外部 Verilog 实现”
4. DSL 仿真遇到 external Verilog module 时，建立协同仿真路径：
   - 顶层 drive / vector / driver 仍以 DSL 模式加载
   - DSL 侧负责 stimulus、时钟/复位与 reference-side orchestration
   - Verilog simulator 负责 external module 所在 RTL 闭环
   - 两边通过统一 trace/端口采样边界对齐
5. 在协同仿真闭环跑通后，补一条 Python-first VIP 路线：
   - stimulus / monitor / scoreboard / reference model 优先支持 Python 编写
   - 优先复用现有 `python_uvm` / `pyuvm` 风格组件，而不是默认落回 SystemVerilog VIP
   - 第一阶段先支持 batch / cycle-aligned 的 Python VIP -> Verilator cosim
   - 第二阶段再扩到更强的 reactive BFM / 协议握手协同
6. Verilog 仿真器路线调整：
   - 新增以 `verilator` 为主的 mixed RTL 仿真/协同仿真能力
   - `run_dsl_rtl_cosim(..., rtl_backend="verilator")` 与
     `run_dsl_multiclock_rtl_cosim(..., rtl_backend="verilator")`
     都要走真实本地编译执行，而不是 mock/fallback
   - 在 `verilator` 之外，补充 `vcs` 作为更接近真实项目交付环境的 cosim backend
   - 文档与计划中不再把 `iverilog` 作为主推荐路径
   - 对已有 `iverilog` smoke/probe/cosim 路径逐步迁移或降级为兼容性路径
7. 增加 mixed-language regression：
   - DSL top + parameterized external Verilog leaf
   - parent wire -> external Verilog -> sibling DSL module
   - single-clock cosim with DSL-driven vectors
   - multi-clock cosim with explicit Verilator execution regression
   - multi-instance parameter override closure
   - Python VIP 驱动 Verilator DUT 的本地闭环 regression
8. 明确 cosim 的性能与交互模型：
   - 当前 cosim 应优先采用“一次生成 TB + 一次外部仿真 + 一次解析 trace”的 batch 模式
   - Python 不应在每个 cycle 都通过进程级 IPC 驱动 `iverilog` / `verilator` / `vcs`
   - `cpp_backend` 继续作为本地高频迭代与 reference-side oracle 的快速路径
   - 后续再评估 persistent external sim / DPI / VPI / socket bridge 以降低重复编译与进程启动开销

### 10.3 验收

1. 用户可以在 DSL 中稳定例化参数化 Verilog module
2. mixed design 的 emitted RTL 不会错误重复生成 external module 壳体
3. DSL 驱动模式下，包含 Verilog leaf 的电路可以走协同仿真闭环
4. 本地推荐 Verilog 仿真器切换为 `verilator`
5. 至少有一条不依赖 SystemVerilog VIP 的 Python VIP -> Verilator 闭环原型
6. `vcs` 已作为可选 cosim backend 接入本地接口层
7. 计划文档中已明确当前 cosim 的性能边界与推荐用法

## 11. 优先级与执行顺序

### 10.1 第一阶段：先修高风险语义坑

1. W1 signed/fixed-point lint 与 regression
2. W2 ownership / flattening regression

1. W4 diagnostics / source-mapped findings
2. W5 cookbook / worked example / README 更新
3. W6 Verilator cosim / external source packaging / migration docs

### 10.3 暂不作为本轮核心目标的事项

以下内容可以继续做，但不作为 `plan0625-v2` 的主考核项：

1. 再引入新的高层 orchestration
2. 自动从 archsim 推 DSL
3. 自动 PPA rewrite
4. 更重的软件栈扩展

## 12.1 W6 当前进展记录

截至本轮推进，W6 已有如下进展：

1. DSL 已支持 external/blackbox Verilog module 的声明、参数覆盖与 artifact 收集
2. emitted RTL 已能跳过 external module 壳体的错误重复生成
3. `run_dsl_rtl_cosim(..., rtl_backend="verilator")` 已接入真实 Verilator 本地执行
4. `run_dsl_multiclock_rtl_cosim(..., rtl_backend="verilator")` 也应按真实执行路径维护与回归
5. Python-first VIP 已有原型：
   - `run_python_uvm_rtl_test(...)` 可用 Python sequence / scoreboard / reference model 检查 Verilator cosim trace
6. 当前 cosim 的效率判断：
   - `cpp_backend` 是进程内 `ctypes` 共享库调用，适合高频本地回归
   - `iverilog` / `verilator` / `vcs` 当前仍是外部进程 compile-run-parse 模式
   - 因此，当前 cosim 更适合作为语义对齐、后端 closure、mixed RTL probe，而不是最高吞吐的主仿真环
7. external simulator cache 已落地到 cosim build 目录复用路径，
   后续 benchmark 应显式区分 cold compile-run 与 warm rerun 两种成本
8. `DslRtlCosimReport` 已开始暴露 cache 元数据：
   - `cache_enabled / cache_hit / cache_key / cache_dir`
   - 上层 benchmark/report 可直接读取命中状态，而不是再扫描目录猜测
9. `benchmark_cosim_cache(...)` 已可量化单 backend 的 cold/warm rerun 收益
10. `benchmark_cosim_backends(...)` 已可汇总 `iverilog` / `verilator` / `vcs`
    的并排对比结果，并保留 skipped_reason / cache / 冷热启动信息
11. backend 对比结果已开始支持 JSON/Markdown 报告输出，
    下一步可直接接真实本地 `verilator` / `vcs` 环境出实测数据
12. 已完成一轮本地真实 backend sweep 试跑：
    - `DslCosimAccum()` 上 `iverilog` 与 `verilator` 均可跑通
    - `verilator` 冷启动约 4.57s，热启动约 0.088s，warm speedup 约 51.9x
    - `iverilog` 冷启动约 0.51s，热启动约 0.107s
    - 当前环境下 `vcs` 仍为 skipped，原因是本地未找到 `vcs`
13. `vcs` backend 已开始支持 remote fallback：
    - 可通过 `RTLGEN_X_REMOTE_VCS_HOST=10.134.143.28`
      与 `RTLGEN_X_REMOTE_VCS_SOURCE_SCRIPT=/apps/EDAs/syn.bash`
      触发 `ssh + source + vcs` 执行路径
    - `auto` 模式和显式 `rtl_backend="vcs"` 都已接入该 fallback
    - 当前 Codex 桌面沙箱内，Python `subprocess(ssh ...)` 仍可能被拦截为
      `Operation not permitted`，因此远端真实闭环可能需要在用户自己的终端环境中运行
14. 用户已完成一轮真实 remote VCS backend sweep：
    - `DslCosimAccum()` 上 `vcs` 已可作为真实 backend 通过 regression
    - 实测 `vcs` cold 约 9.17s，warm 约 3.39s，说明当前 remote VCS 热启动瓶颈
      主要不在编译缓存，而在 SSH / upload / remote process startup 往返成本
    - 下一步优化重点应放在减少每次 run 的 SSH 次数，或引入 persistent remote session

## 12.2 W3/W4 当前进展记录

截至本轮推进，DSL 本身的 storage/diagnostics 收口又补了一块明确边界：

1. `VerilogEmitter` 对 `Memory(...)` 的 emitted RTL 支持范围已统一收紧为显式 contract：
   - 当前只正式支持 `read_ports=1`、`write_ports=1`、
     `read_style="async"`、`read_latency=0`
   - 对 `read_style="sync"`、非零 `read_latency`、多读口/多写口 metadata
     不再依赖 generic `NotImplementedError`，而是统一抛 `DslLoweringError`
2. 新的 storage emit fail-fast diagnostics 现在会直接带出：
   - module 名
   - memory 名
   - 具体越界 contract 项（如 `read_style='sync'`、`read_latency=1`、`read_ports=2`）
   - 推荐路径：走 executable lowering/cosim，或显式作者自己写 sampled-output / multi-port RTL
3. `emit_design(...)` 也补成了先对 dedup 后的所有 DSL submodule 做 preflight 校验，
   因此子模块里若含不支持的 storage contract，会在真正吐 RTL 之前 fail-fast，
   不再出现“先输出部分模块，再中途炸掉”的边界行为
4. 已补 regression：
   - single-module sync-read memory emit fail-fast
   - multi-port memory metadata emit fail-fast
   - `emit_design(...)` 对 child storage contract 的 preflight fail-fast
5. 这一步的意义不是扩大 emitted RTL 的 storage 能力面，而是先把 W3/W4 的边界讲清楚：
   - executable lowering 已能接住 `sync-read/read_latency=1`
   - emitted RTL 仍然只承诺 1R/1W async-zero-latency 子集
   - 超出子集时，框架现在会更早、更明确地把作者带回推荐写法

## 12.3 W2/W4 当前进展记录

截至本轮推进，层次化 authoring 的一个高风险 silent-failure 点也已补成 fail-fast：

1. 显式 `instantiate(..., port_map=...)` 的 `port_map` key 现在会在 public boundary
   上做合法性校验：
   - 每个 key 都必须匹配 child module 的真实 input/output port 名
   - 不再允许写错端口名后被 flatten/lower 路径静默忽略
2. 新增 diagnostics：
   - `[UnknownSubmodulePort]`
   - 直接给出 instance path、错误端口名、child module 名、以及 valid ports 列表
3. flatten 路径也补成了同一语义：
   - 过去 unknown port 在 `_inline_submodule(...)` 中会 `continue`
   - 现在会明确抛错，而不是把这条连接悄悄丢掉
4. 已补 regression：
   - `validate_authoring_intent(...)` 对 unknown submodule port fail-fast
   - `lower_dsl_module_to_sim(...)` 对 unknown submodule port fail-fast
   - `VerilogEmitter().emit(...)` 对 unknown submodule port fail-fast
5. 这一步主要收口的是 W2/W4 的 authoring contract，而不是扩展功能面：
   - 合法层次化连接继续保持 happy path
   - 非法 `port_map` typo 不再演化成后续 flatten/emit 的静默错误

## 12.4 W2 当前进展记录

截至本轮推进，W2 又补上了一条此前只在 emitter 侧“局部修补”、但在 lowering 侧并未真正闭环的合法 happy path：

1. 隐式子模块 + parent-owned interconnect 现在已在 lowering / flatten / emit 三条路径统一收口：
   - 典型写法：
     - `self.u_leaf = Child()`
     - `self.u_leaf.clk <<= self.clk`
     - `self.mid <<= self.u_leaf.y`
   - 这类写法现在会先统一归一化为明确的 submodule `port_map` 与 helper wire，再进入 flatten / lowering / emitted RTL
2. 过去的实际问题是：
   - emitter 侧有一套独立的 cross-module assignment 修补
   - lowering/flatten 侧没有同样的归一化
   - 结果是合法 parent->child input 连线会在 lowering 时被误当成“组合块里给 input 赋值”，直接报错
3. 现在的修复是：
   - lowering 入口先做 shared cross-module normalization
   - emitter 也改为复用同一套 normalization，而不再保留独立修补逻辑
   - child output 经 parent wire 中转时，会显式生成稳定 helper wire（如 `u_leaf_y`）并在两个后端上保持一致语义
4. 在这条 happy path 跑通后，又顺手暴露并修掉了一个 comb assignment ordering 的内部 bug：
   - 旧逻辑默认 assignment 原始位置是 `0..N-1` 连续编号
   - 新的层次化归一化后，source-mapped ordering 会出现稀疏位置
   - 现已改为按真实已收集位置重排，不再触发内部 `KeyError`
5. 已补 regression：
   - `lower_dsl_module_to_sim(QueryTop())` 支持隐式子模块 parent interconnect
   - `VerilogEmitter().emit(QueryTop())` 对同一结构保持一致 helper-wire 连接
   - 相关 hierarchy / connectivity report 现有回归保持通过

## 12.5 W2/W3 当前进展记录

截至本轮推进，audit 里提到的 “submodule 内部 `Array` 读取最好先手动过一个 wire，否则 flatten 后容易掉” 这类经验，也开始被 DSL 自己收进正式 happy path：

1. 现在对于隐式子模块输出直接承载 storage read 的模式，例如：
   - child 内部 `self.dout <<= self.rf[self.addr]`
   - parent 侧 `self.u.addr <<= self.addr`、`self.out <<= self.u.dout`
   shared cross-module normalization 会先把它规整成稳定的 `port_map + helper wire`
2. 这意味着下面两类 through-submodule 结构现在都已有 regression 覆盖：
   - `ArrayRead` through implicit submodule
   - `MemRead` through implicit submodule
3. 在 lowering 侧：
   - 会生成明确的 helper signal（如 `u_dout`）
   - 会把 child 内部存储对象稳定 rename 成 `u_rf` / `u_mem` 等 flatten-safe 名称
4. 在 emitted RTL 侧：
   - submodule 例化会显式连到 helper wire
   - parent 输出再从 helper wire 取值
   - 不再要求作者手动先在 child 内部再过一层“为 flatten 服务”的临时 wire
5. 这一步还不是对所有层次化 storage 模式的最终收口，但它已经把 audit 中最典型的一类 datapath 写法从“靠经验避坑”推进到了“框架主动兜住”的阶段

## 12.6 W2/W3 当前进展记录

截至本轮推进，层次化 interconnect 的 happy path 又从“child output 直接接 parent wire”继续扩到“child output 可直接进入 parent 组合表达式”：

1. shared cross-module normalization 现在不只会处理：
   - `self.out <<= self.u.y`
   还会递归改写下面这类 parent 侧表达式：
   - `self.out <<= SRA(self.u.y, 2).as_uint()[7:0]`
   - 以及同类的 slice / concat / mux / compare 中嵌入 child output 的写法
2. 修复前的问题是：
   - 只有“裸 child output”会被 helper-wire 化
   - 一旦 child output 出现在 parent 表达式内部，lowering 会残留跨层引用，语义和 flatten/emit 路径不一致
3. 这轮同时把层次化 clone/flatten 对信号声明元信息的复制补齐了：
   - helper wire 会保留 `signed`
   - flatten 后的 `u_*` wire/reg 也会保留 `signed`
   - 避免 parent 在 helper wire 上继续做算术/移位时 silently 退化成 unsigned 语义
4. 已补 regression：
   - `child output in parent expression` 的 lowering/runtime 回归
   - 同结构 emitted RTL 回归，确认 helper wire 与 `$signed(...)/>>>` 语义保持一致
5. 这一步的意义是把 W2/W3/W4 交叉处一个很常见的 datapath 写法真正收口：
   - 作者不必再把 child output 先手动落到 parent 临时 wire 再参与表达式
   - DSL 对层次化 signed datapath 的支持，从“结构可连”推进到了“表达式也能稳”

## 13. 计划完成标准

`plan0625-v2` 完成，不是指“写完计划”，而是至少达到下面结果：

1. JPEG 审计里暴露的三类高频框架问题
   - signed trap
   - hierarchical interconnect fragility
   - storage/flatten fragility
   已经有明确代码修复或明确 fail-fast 诊断
2. Python / C++ / emitted RTL 的关键语义一致性回归更完整
3. DSL 作者不再需要频繁依赖隐式内部约定
4. 文档里已经把推荐写法和未支持边界讲清楚
5. DSL/Verilog 混合设计已经有明确 authoring 入口与本地闭环路线

---

## 14. 总结

这次 JPEG 设计给我们的最重要结论不是“框架不行”，而是：

1. 当前 `rtlgen_x` 已经足以支撑真实 datapath 设计
2. 但它还没有把最常见的 datapath 风险收紧成稳定 authoring surface
3. 下一步最值得做的，不是继续铺更宽的功能面，而是把 signed、层次化、存储、诊断这几条主路径做扎实

一句话收束：

**先把真实设计里最容易卡手的地方修到顺手，再继续扩展框架边界。**
