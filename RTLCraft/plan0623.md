# Plan 0623: `rtlgen_x` DSL / Verilog / ArchSim 收敛与增强计划

## 1. 背景

基于当前 `rtlgen_x` 的真实代码状态，以及我们刚完成的一轮审视，这里需要把几个关键判断正式收敛成新的执行计划：

1. `rtlgen_x.dsl` 现在已经是唯一主 authoring surface
2. 多时钟能力在语义上基本可用，但 authoring 体验还不够强
3. DSL 输出的 Verilog/SystemVerilog 已经能工作，但距离“稳定、优雅、适合 review 的 RTL”还有差距
4. `archsim` 适合作为轻量架构探索与报告工具，但还不应该被宣传成高保真复杂 SoC 性能模型

这份计划不是替代 [plan_dsl_0623.md](/Users/yangfan/release/EDACraft-main/RTLCraft/plan_dsl_0623.md)，而是把本轮更具体的开发方向、执行顺序和验收口径补齐。

---

## 2. 总体目标

这一轮工作的目标是把 `rtlgen_x` 收紧成一个更清晰、更可信、也更适合 agent 使用的硬件能力栈：

1. **DSL 更像工程语言，而不是“Python 版 RTL 拼接器”**
2. **多时钟 / reset / CDC 能力在 authoring 层表达清楚**
3. **生成 RTL 的可读性、稳定性、review 质量明显提升**
4. **archsim 的定位、边界、适用方式写清楚，并补强最有价值的探索能力**

一句话说：

**让 `rtlgen_x` 在“写得出、跑得通、看得清、分析得准、方便 agent 改”这五件事上一起变强。**

---

## 3. 明确边界

本计划明确保持以下边界，不再回到过去的重控制面模式：

1. `archsim` 只输出报告，不显式向 DSL 传递结构化 IR
2. `ppa` 只输出分析和建议，不自动改写 DSL
3. 不恢复第二套 DSL front-end
4. 不恢复多层文档 IR 控制流
5. 不引入新的 repo-wide contract / orchestrator
6. agent 负责组织这些能力，`rtlgen_x` 只提供强硬能力

---

## 4. 当前判断

### 4.1 DSL 现状

当前 DSL 已经具备真实价值，原因不在于它“更像 Verilog”，而在于它有：

1. 程序化参数化能力
2. executable-first 语义
3. lowering 到统一执行模型的能力
4. 协议 / bundle / 结构化抽象
5. 对验证、PPA、CDC、agent 更友好的结构化入口

但它的问题也很明确：

1. 类型系统还弱
2. channel / protocol 语义还不够一等公民
3. 多时钟语法可写，但不够自然
4. 某些设计 intent 仍然需要用户“知道内部约定”才能写稳

### 4.2 多时钟现状

当前多时钟路径在 lowering、Python 仿真、编译仿真、Python-UVM、部分 cosim 上已经可以工作，但还有明显短板：

1. domain 绑定写法偏显式、偏啰嗦
2. crossing intent 缺少更高层 authoring primitive
3. reset-release 诊断还可以继续强化
4. 多时钟 UVM 使用方式还偏 directed / specialist-friendly

### 4.3 Verilog 输出现状

当前 emitter 输出已经比“纯机器拼接文本”要好，能做到：

1. `always_comb` / `always_ff` / `always_latch`
2. 基本结构稳定
3. profile 化输出
4. 与 DSL source mapping / review profile 有一定结合

但还不够：

1. 大模块命名稳定性不足
2. 中间信号和 temporary 的组织还不够好
3. 组合表达式有时过于扁平
4. 缺少“可读性 golden RTL 回归”的硬门槛

### 4.4 archsim 现状

`archsim` 当前最合适的定位是：

1. 轻量级、通用的架构探索器
2. 支持 behavior-level 与 cycle-level 两层
3. 适合 CPU/GPGPU/NPU/控制器/通用数据通路的早期建模
4. 输出 bottleneck、capacity、latency、bandwidth、queue-depth 等报告

但它现在还不应该承担这些承诺：

1. 复杂 SoC 的自动高保真性能预测
2. 从 RTL/DSL 自动推断完整微架构
3. 细粒度仲裁、冲突、层次化共享资源的真实还原

## 5. 执行总路线

本轮按照四条主线推进：

1. **W1: DSL 语义与 authoring 能力增强**
2. **W2: 多时钟 / reset / CDC 能力增强**
3. **W3: Verilog 可读性与 emitter 质量增强**

3. 有回归测试
4. 有清晰验收口径

## 6. W1: DSL 语义与 authoring 能力增强

### 6.1 目标

让 DSL 的强项更明显，避免它退化成“另一种写法的 Verilog”。

### 6.2 重点方向

#### A. 类型系统增强

优先级从高到低：

1. `enum` 状态类型
2. packed `struct` / field access
3. fixed-point 类型
4. lane-vector / SIMD-friendly 类型

目标不是追求语法花哨，而是让真实控制器、协议头、向量 datapath 更自然。

#### B. channel / protocol 一等公民化

把下列对象从“bundle + helper”推进成更统一的 DSL 语义对象：

1. ready-valid channel
2. request-response channel
3. CSR / register-bank interface
4. streaming packet / framed stream channel

需要配套：

1. protocol lint
2. auto-connect helper
3. adapter / width-shim / clock-crossing shim
4. Python-UVM / SV-UVM 消费接口

#### C. 设计 intent 更显式

继续减少“靠经验知道怎么写才安全”的部分，补强：

1. register intent
2. state intent
3. memory intent
4. handshake intent
5. pipeline stage intent

### 6.3 主要改动点

1. `rtlgen_x/dsl/core.py`
2. `rtlgen_x/dsl/logic.py`
3. `rtlgen_x/dsl/protocols.py`
4. `rtlgen_x/dsl/pipeline.py`
5. `rtlgen_x/dsl/lib.py`
6. `rtlgen_x/dsl/adapter.py`

### 6.4 验收标准

1. 至少一套 `enum`/`struct` 新类型能完成 DSL -> lowering -> sim -> emit 闭环
2. 至少一套 channel 语义对象能完成 DSL -> verify/UVM -> emit 闭环
3. 新特性在 `README` 和 support matrix 中有明确状态标注

---

## 7. W2: 多时钟 / reset / CDC 能力增强

### 7.1 目标

让多时钟设计不是“底层能跑”，而是“用户在 DSL 层能清楚写、工具能清楚查、验证能清楚测”。

### 7.2 重点方向

#### A. Domain authoring ergonomics

补强或收紧：

1. `clock_domain(...)` / `seq_domain(...)` 的使用方式
2. process 与 domain 的显式绑定
3. module 级 domain registry
4. active domain 推导和缺失诊断

目标是减少用户在多时钟设计里手工传 `active_domains` 的负担。

#### B. reset 语义收紧

继续加强：

1. async assert / sync release 的显式声明
2. reset polarity 推断与冲突报错
3. reset-release 专项 CDC 规则
4. reset builder / synchronizer pattern 识别

#### C. CDC authoring primitive

优先支持和标准化：

1. 2-flop synchronizer
2. pulse synchronizer
3. ready-valid async bridge
4. async FIFO wrapper
5. reset synchronizer

这里的目标不是“所有 crossing 自动修复”，而是：

1. 手工写法能被识别
2. 标准原语有稳定实现
3. 报告能明确指出 crossing 风险和建议

### 7.3 主要改动点

1. `rtlgen_x/dsl/core.py`
2. `rtlgen_x/dsl/lib.py`
3. `rtlgen_x/verify/cdc.py`
4. `rtlgen_x/verify/uvm.py`
5. `rtlgen_x/verify/python_uvm.py`
6. `rtlgen_x/sim/python_runtime.py`
7. `rtlgen_x/sim/cpp_backend.py`

### 7.4 验收标准

1. 单时钟、多时钟、reset-release 都有清晰 DSL authoring 示例
2. CDC 报告能稳定回指 module / signal / domain / source line
3. hand-written synchronizer pattern 识别覆盖明显提升
4. 多时钟设计至少有一组 Python sim / C++ sim / UVM 闭环测试

## 8. W3: Verilog 可读性与 emitter 质量增强

### 8.1 目标

让 emitted RTL 不只是“可综合、可仿真”，还要：

1. 更接近可维护手写 RTL
2. 更适合 code review
3. 更适合 agent 二次修改

### 8.2 重点方向

#### A. 命名稳定性

继续提升：

1. 中间 wire / reg 名称稳定性
2. bundle / field 展开后的命名一致性
3. stage / pipeline / protocol 相关信号的成组命名
4. temporary name 避免出现“机器味”过重的散乱编号

#### B. 结构化输出

优化 emitter 的组织策略：

1. port / parameter / localparam / type definition 分区
2. combinational / sequential / latch block 分区
3. memory / array / helper logic 分区
4. protocol / channel 接口相关逻辑适度聚合

#### C. 表达式可读性

补强：

1. 复杂表达式拆分
2. ternary 嵌套限制
3. 宽位切片、拼接、符号扩展的规范化
4. 适量 comment / source map / section heading

#### D. golden RTL gating

引入可读性回归门槛：

1. 核心标准库模块 golden RTL snapshot
2. 多时钟模块 golden RTL snapshot
3. protocol / register bank / FIFO / pipeline 模块 golden RTL snapshot
4. emitter profile 的稳定性测试

### 8.3 主要改动点

1. `rtlgen_x/dsl/codegen.py`
2. `rtlgen_x/dsl/emitter.py`
3. `rtlgen_x/dsl/adapter.py`
4. `rtlgen_x/tests/` 下的 emitted RTL snapshot tests
5. `rtlgen_x/README.md`
6. `rtlgen_x/DSL_SUPPORT_MATRIX.md`

### 8.4 验收标准

1. 典型模块 emitted RTL 的可读性明显提升
2. snapshot 测试能防止 emitter 风格回退
3. 多时钟 / protocol / stdlib 模块的 RTL 输出结构稳定
4. 文档明确说明推荐 emitter profile 和 review 使用方式

## 9. W4: archsim 定位收敛与能力补强

### 9.1 目标

把 `archsim` 做成一个轻量但靠谱的架构探索工具，而不是一个模糊承诺太多的“万能自动建模器”。

### 9.2 核心定位

`archsim` 的主价值是：

1. 快速建模
2. 轻量 sweep
3. 报告 bottleneck
4. 比较 capacity/latency/II/bandwidth 的折中
5. 支持 CPU/GPGPU/NPU/控制器/数据通路的通用场景

它不负责替代详细 RTL 仿真，也不负责自动生成 DSL。

### 9.3 重点方向

#### A. 强化手工建模路径

优先把这些能力做强：

1. `ArchitectureModel` authoring API
2. reusable stage builders
3. composite scenario helpers
4. parameter sweep helpers
5. report summarization / markdown export

#### B. 弱化自动推断承诺

对于 `infer_architecture_from_module(...)` 一类功能：

1. 文档里明确标成 heuristic / early-estimate
2. 不再暗示它能自动恢复真实微架构
3. 报告中显式标注推断置信度或限制

#### C. 补强通用架构维度

优先考虑：

1. hierarchical/shared resource contention
2. queue/backpressure propagation
3. memory hierarchy pressure summary
4. compute array occupancy / utilization summary
5. behavior-level 与 cycle-level 结果对比

#### D. 形成推荐使用流程

推荐的架构评估工作流应明确为：

1. 手工建模或在 reference scenario 基础上改
2. 跑 behavior-level 快速扫参数
3. 对重点方案跑 cycle-level
4. 输出 bottleneck / upgrade ranking / markdown report
5. 由 agent 决定是否回改 DSL/RTL

### 9.4 主要改动点

1. `rtlgen_x/archsim/models.py`
2. `rtlgen_x/archsim/simulator.py`
3. `rtlgen_x/archsim/infer.py`
4. `rtlgen_x/archsim/report.py`
5. `rtlgen_x/archsim/examples.py`
6. `rtlgen_x/TUTORIAL_ARCH_PPA.md`

### 9.5 验收标准

1. 文档对 `archsim` 的定位、输入方式、边界写清楚
2. reference scenario 覆盖 CPU/GPU/NPU/controller/datapath 五类典型场景
3. behavior-level / cycle-level 都有稳定示例和回归
4. 自动推断能力的 heuristic 边界在报告和文档中明确可见

## 11. 执行优先级

建议按以下顺序推进：

### P0

1. 新增 `plan0623.md`
2. 在文档中明确 DSL / archsim / emitter 的当前定位
3. 把 `infer_architecture_from_module(...)` 的 heuristic 边界写清楚

### P1

1. 先补多时钟 / reset / CDC authoring 与诊断
2. 同时推进 Verilog emitter 的命名与结构化输出

### P2

1. 再补 DSL 类型系统和 protocol/channel 语义增强
2. 建立 golden RTL snapshot 回归

### P3

1. 最后继续补强 archsim 的层次化建模与报告
2. 完成文档、tutorial、example 的整体验收

原因很简单：

1. 多时钟和 emitter 质量最直接影响真实设计可用性
2. 类型系统和协议增强应建立在更稳的语义与输出基础上
3. archsim 需要先把定位收紧，再逐步补能力，避免误用

## 12. 最终验收口径

当以下条件同时满足时，可以认为这一轮 `0623` 计划达到阶段性目标：

1. 用户能清楚知道 DSL 的优势、边界和推荐写法
2. 多时钟 / reset / CDC 设计有稳定的 authoring + 检查 + 测试路径
3. emitted RTL 达到更稳定、更适合 review 的质量
4. `archsim` 的适用场景、局限、推荐流程都清晰可见
5. 所有增强能力都有测试和文档，而不是只停留在零散实现

---

## 13. 下一步建议

这份计划落地后，后续实现建议优先从两件事开始：

1. **先做 W2：多时钟 / reset / CDC**
2. **并行做 W3 的 emitter 可读性收口**

因为这两项最直接决定：

1. DSL 是否能安全地写真实系统
2. 生成 RTL 是否真的能被人接住和维护

等这两块站稳，再去放大类型系统、协议语义和 archsim 的建模深度，会更顺。
