对，你这个理解更准确。你的目标不是：

1. **直接复用 skills 里的 DSL 模块**；
2. 也不是只把 skills 当普通文本参考；
3. 而是：

> **骨架生成器提出“我需要什么模块/功能”，系统到 skills 里检索相似 DSL 模块，把它们作为参考样例，然后驱动 agent 根据当前骨架需求生成新的实际 DSL 代码，并接入行为级模型做验证与修复。**

也就是说，完整目标是：

```text
行为级模型 / 系统级需求
        ↓
骨架生成需要的模块需求
        ↓
检索 skills/dsl_modules.py 中相关参考模块
        ↓

        ↓
Agent 生成当前专用 DSL 代码
        ↓
DSL → RTL
        ↓
行为级模型 / golden trace 验证
        ↓
失败则 agent 修复
```

这应该叫：

# Skeleton-Requirement + Skill-Reference + Agent DSL Generation

# Skill-Referenced DSL Generation Flow

# 1. 你的框架应重新分工

当前框架里几个部分的职责应该改成：

| 组件                  | 职责                                                  |
| --------------------- | ----------------------------------------------------- |
| 行为级模型            | 定义功能正确性，产生 golden trace / expected behavior |
| ArchSkeletonGenerator | 生成模块需求，而不是直接生成完整逻辑                  |
| SkillRetriever        | 从 skills 中找相似 DSL 模块/片段                      |
| ReferenceExtractor    | 从参考 DSL 中提取结构、接口、状态机、关键逻辑         |
| Agent DSL Generator   | 根据骨架需求 + 参考 DSL 生成新的 DSL 代码             |
| DSL Compiler          | 将 DSL 转为 RTL                                       |
| Verifier              | 用行为级模型或 golden tests 验证生成结果              |
| Repair Agent          | 根据错误报告修复 DSL                                  |

最关键的是：

> ArchSkeletonGenerator 的输出不应只是“空模块”，而应该是 **ModuleRequirement**。

# 2. 核心数据结构：ModuleRequirement

骨架生成器不要直接问：

```text
我要生成 sm_wrapper
```

而是输出结构化需求：

```yaml
module_requirement:
  name: sm_0
  type: sm_wrapper
  role: gpgpu_streaming_multiprocessor

  parameters:
    NUM_WARP: 8
    NUM_THREAD: 32
    DATA_WIDTH: 32
    NUM_REG: 256
    IBUF_DEPTH: 4

  ports:
    clk:
      dir: input
      type: logic
    rst_n:
      dir: input
      type: logic
    dispatch_valid:
      dir: input
      type: logic
    dispatch_ready:
      dir: output
      type: logic
    dispatch_payload:
      dir: input
      type: logic[127:0]
    mem_req_valid:
      dir: output
      type: logic
    mem_req_ready:
      dir: input
      type: logic
    mem_req_addr:
      dir: output
      type: logic[31:0]
    mem_rsp_valid:
      dir: input
      type: logic
    mem_rsp_rdata:
      dir: input
      type: logic[31:0]

  required_behaviors:
    - receive CTA dispatch through valid-ready handshake
    - allocate warp slots
    - maintain per-warp PC and active mask
    - fetch instruction from instruction memory
    - decode opcode, rs1, rs2, rd, imm
    - schedule one ready warp per cycle
    - check scoreboard dependency before issue
    - execute ALU operations
    - generate load/store memory request
    - write back ALU/load result to VRF
    - handle barrier synchronization

  internal_state_suggestions:
    - warp_valid[NUM_WARP]
    - warp_pc[NUM_WARP]
    - warp_mask[NUM_WARP]
    - scoreboard_busy[NUM_REG]
    - ibuf_fifo
    - rr_ptr

  verification_hooks:
    - trace_issue_warp_id
    - trace_writeback_valid
    - trace_writeback_rd
    - trace_writeback_data
    - trace_mem_req_valid
```

这个 `ModuleRequirement` 是后续检索、生成、验证的中心。

# 3. skills 检索目标：找“参考 DSL 模块”，不是找可直接复用模块

检索输入是 `ModuleRequirement`。

检索输出不是完整代码，而是：

```yaml
reference_modules:
  - name: SMWrapper
    file: skills/gpgpu/dsl_modules.py
    relevance: 0.91
    why_relevant:
      - same architectural role: streaming multiprocessor wrapper
      - contains warp scheduler, scoreboard, issue, execute, writeback flow
    useful_parts:
      - submodule organization
      - warp FSM
      - memory request flow
      - barrier logic
    caution:
      - interface names differ from current skeleton
      - some parameters are hardcoded

  - name: WarpScheduler
    file: skills/gpgpu/dsl_modules.py
    relevance: 0.86
    useful_parts:
      - round-robin warp selection
      - ready mask generation
      - issue_warp_id selection

  - name: Scoreboard
    file: skills/gpgpu/dsl_modules.py
    relevance: 0.82
    useful_parts:
      - busy-bit dependency tracking
      - writeback clears busy bits
      - issue blocked on hazard

  - name: IBuffer
    file: skills/gpgpu/dsl_modules.py
    relevance: 0.74
    useful_parts:
      - valid-ready FIFO interface
      - instruction buffering
```

检索的关键不是名字匹配，而是根据：

```text
行为相似性
接口相似性
控制模式相似性
状态组织相似性
数据通路相似性
验证目标相似性
```

------

# 4. ReferenceExtractor：从参考 DSL 中提取有用信息

直接把整个 `SMWrapper` 代码塞给 agent，容易导致复制、污染和超长上下文。

更好的做法是提取参考摘要：

```yaml
reference_summary:
  module: WarpScheduler

  design_intent:
    - select one warp among multiple active warps
    - avoid issuing blocked warps
    - preserve fairness through round-robin pointer

  interface_pattern:
    inputs:
      - warp_valid vector
      - scoreboard_ready vector
      - issue_ready
    outputs:
      - issue_valid
      - issue_warp_id

  state_pattern:
    - rr_ptr register

  logic_pattern:
    combinational:
      - compute ready_mask = warp_valid & scoreboard_ready
      - scan entries from rr_ptr
      - choose first ready warp
    sequential:
      - update rr_ptr on issue_fire

  pseudo_dsl:
    - Reg rr_ptr
    - Wire ready_mask
    - comb loop selects issue_warp_id
    - ff block updates rr_ptr

  adaptation_notes:
    - rename scoreboard_ready to dependency_clear
    - use current skeleton NUM_WARP parameter
    - connect issue_fire = issue_valid & issue_ready
```

这样 agent 得到的是 **结构化参考**，不是被迫照抄旧代码。

# 5. Agent 生成输入：GenerationContext

最终喂给 agent 的上下文应该是：

```yaml
generation_context:
  target:
    module_requirement: ...
  
  reference_dsl_modules:
    - reference_summary: SMWrapper
    - reference_summary: WarpScheduler
    - reference_summary: Scoreboard
    - reference_summary: IBuffer

  generation_task:
    output_language: RTLCraft_DSL
    generate_complete_module: true
    keep_ports_exactly_as_required: true
    use_reference_only_as_guidance: true
    do_not_copy_reference_code_verbatim: true
    synthesizable_only: true

  coding_rules:
    - use explicit registers for state
    - use always_ff for sequential state
    - use always_comb for combinational outputs
    - no latches
    - no multiple drivers
    - reset all state registers
    - use valid-ready fire condition for state updates

  verification_contract:
    - after reset, dispatch_ready should be high
    - when dispatch_valid and dispatch_ready, allocate a warp
    - ready warp should eventually issue
    - ALU instruction should write back expected result
    - load instruction should generate memory request then write back response
```

这一步才是真正的“agent 生成”。

# 6. 生成方式：不要一次生成整个复杂模块

复杂模块如 SMWrapper，建议分两层生成：

## 第一层：生成模块结构

Agent 先生成 DSL module 的整体框架：

```text
ports
parameters
internal state
submodule declarations
pipeline channels
trace hooks
```

## 第二层：按 task 生成逻辑

每个 task 生成一部分 DSL：

```text
Task 1: dispatch_to_warp
Task 2: instruction_fetch
Task 3: decode
Task 4: warp_schedule
Task 5: scoreboard
Task 6: issue
Task 7: execute_alu
Task 8: load_store
Task 9: writeback
Task 10: barrier
```

每个 task 都有自己的 reference modules。

例如：

```yaml
task: warp_schedule
references:
  - WarpScheduler
  - SMWrapper scheduling part
task: scoreboard
references:
  - Scoreboard
  - Writeback
task: load_store
references:
  - LSU
  - memory_request_pattern
```

这样可以避免一次生成失控。

# 8. 行为级模型如何接入验证

行为级模型不要只在最开始仿真通过，还要作为 RTL 生成后的 golden。

建议输出两类东西。

## 8.1 Golden Trace

行为模型运行一个测试程序，输出 trace：

```json
[
  {
    "cycle": 0,
    "event": "dispatch",
    "cta_id": 0
  },
  {
    "cycle": 3,
    "event": "issue",
    "warp_id": 0,
    "opcode": "ADD"
  },
  {
    "cycle": 5,
    "event": "writeback",
    "warp_id": 0,
    "rd": 1,
    "data": 29
  }
]
```

RTL 仿真也输出类似 trace，然后比较：

```text
行为模型 trace ≈ RTL trace
```

注意不一定要求 cycle 精确，第一版可以做 event-order equivalence：

```text
dispatch → issue ADD → writeback rd=1,data=29
```

第二版再做 cycle-aware。

------

## 8.2 Property / Assertion

行为模型还可以导出局部 property：

```yaml
properties:
  - name: no_issue_invalid_warp
    rule: issue_valid implies warp_valid[issue_warp_id]

  - name: scoreboard_blocks_hazard
    rule: if scoreboard_busy[rs1], instruction cannot issue

  - name: load_generates_mem_request
    rule: load opcode eventually causes mem_req_valid
```

这些 property 可以转成：

```text
SystemVerilog assertion
Python simulation checker
DSL-level checker
```

# 10. Repair Loop

生成失败后，不应该重新从头生成，而应该局部修复。

Repair 输入：

```yaml
repair_context:
  module_requirement: ...
  generated_dsl: ...
  failed_stage: rtl_simulation
  error_log: ...
  behavior_trace_diff:
    expected:
      - writeback rd=1 data=29
    actual:
      - no writeback observed
  relevant_references:
    - WarpScheduler
    - Writeback
    - Scoreboard
  instruction:
    - make minimal DSL patch
    - do not change module ports
    - preserve existing working logic
```

这样 agent 才能迭代收敛。

# 11. 整体主流程伪代码

```python
def generate_rtl_with_skill_references(behavior_model, arch_def, skills_db):
    # 1. 行为级仿真
    behavior_result = behavior_model.run_tests()
    assert behavior_result.passed

    golden_traces = behavior_result.traces
    behavior_contracts = behavior_model.export_contracts()

    generated_modules = []

    # 2. 骨架生成模块需求
    module_requirements = ArchSkeletonGenerator.extract_requirements(
        arch_def=arch_def,
        behavior_contracts=behavior_contracts
    )

    for req in module_requirements:
        # 3. 为模块生成任务列表
        tasks = build_generation_tasks(req)

        # 4. 逐任务检索参考 DSL
        task_contexts = []
        for task in tasks:
            query = build_skill_query(req, task)
            refs = SkillRetriever(skills_db).retrieve(query, top_k=5)
            summaries = ReferenceExtractor.extract(refs, task)
            task_contexts.append((task, summaries))

        # 5. 构造 agent 生成上下文
        gen_ctx = GenerationContext(
            requirement=req,
            task_contexts=task_contexts,
            behavior_contract=behavior_contracts.get(req.name),
            generation_policy={
                "output": "DSL",
                "synthesizable": True,
                "do_not_copy_reference_verbatim": True
            }
        )

        # 6. Agent 生成 DSL
        dsl_module = AgentDSLGenerator.generate(gen_ctx)

        # 7. 静态检查 + 编译 + 行为验证
        for attempt in range(MAX_REPAIR):
            check = Verifier.check_dsl(dsl_module)
            if not check.passed:
                dsl_module = AgentDSLGenerator.repair(dsl_module, check, gen_ctx)
                continue

            rtl = DSLCompiler.compile(dsl_module)

            sim = Verifier.run_rtl_sim(rtl, golden_traces.get(req.name))
            if sim.passed:
                break

            dsl_module = AgentDSLGenerator.repair(dsl_module, sim, gen_ctx)

        generated_modules.append(dsl_module)

    # 8. 顶层集成
    top_dsl = integrate_generated_modules(generated_modules, arch_def)

    # 9. 全系统验证
    final_rtl = DSLCompiler.compile(top_dsl)
    final_check = Verifier.verify_system(final_rtl, golden_traces)

    return final_rtl, final_check
```

# 12. 你当前框架需要怎么改

## 12.1 `ArchSkeletonGenerator` 不直接填完整逻辑

它应主要输出：

```text
ModuleRequirement
TaskList
PortSpec
StateSuggestion
VerificationHook
```

而不是无条件调用 `_fill_sm_wrapper_logic`。

原 `_fill_sm_wrapper_logic` 可以保留为：

```text
fallback reference implementation
```

但不能作为主路径。

------

## 12.2 `skills/dsl_modules.py` 不直接绑定实例化

你这次明确说了，不是直接用现成代码。所以 skills 的用途是：

```text
Reference DSL source
```

需要增加：

```text
SkillRetriever
ReferenceExtractor
```

而不是只做 `SkillBinder`。

------

## 12.3 `LogicGenerator` 要升级为 AgentDSLGenerator

当前 `LogicGenerator` 是规则模板。

你需要让它能接受：

```text
ModuleRequirement
Reference DSL summaries
Behavior contract
Error feedback
```

然后生成完整 DSL。

也就是说从：

```text
pattern emitter
```

变成：

```text
context-aware DSL synthesis agent
```
