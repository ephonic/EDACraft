# Plan 0622: Archsim / Multi-Clock / Area-Power PPA

## 背景

基于 `gpu_sm` 的再次验证与 [audit0621-kimi.md](/Users/yangfan/release/EDACraft-main/RTLCraft/audit0621-kimi.md)，当前 `rtlgen_x` 的核心能力已经可用，但还有三类问题需要进一步收口：

1. 架构仿真能力是否足够，设计中应如何使用，并如何以报告方式支撑设计决策
2. DSL 层是否可以支持多时钟设计，哪些路径是真支持，哪些路径当前不能闭环
3. PPA 分析除时序外，对面积和功耗如何建模、如何校准、如何给 agent 更具体的建议

本计划只围绕“硬能力”推进，不重新引入重型控制面或多层流程框架。

## 前置决策

### D0. `seq` 中直接驱动 `Output` 暂不做自动 shadow-reg 改写

当前保留“清晰报错 + 推荐写法”的策略，暂不做 lowering 时的隐式 auto shadow-reg。

原因：

1. 自动改写会隐式改变端口可观察时序
2. reset 语义、命名、UVM 观测点和调试路径都会被悄悄改动
3. 对 framework 来说，显式 `Reg -> comb out` 比隐式修复更可控

后续如确有必要，只考虑增加**显式 helper**，不做静默自动修复。

## 总体目标

在不增加重型流程框架的前提下，把 `rtlgen_x` 的以下三件事补强到“可直接被 agent 使用”的状态：

1. 架构仿真结果可以稳定形成报告并指导 DSL/RTL 结构决策
2. 多时钟支持边界清晰，单时钟/多时钟各自有明确可用路径
3. PPA 的面积/功耗分析从“只有启发式”提升到“启发式 + 热点定位 + 可校准”的报告能力

## 执行顺序

按下面顺序推进：

1. 先补 support matrix / guardrail，避免误用
2. 再补 archsim 报告化使用方式
3. 再补面积/功耗 PPA 的可解释性和校准路径
4. 最后推进新 runtime 的多时钟闭环支持

这样做的原因是：先把边界画清楚，再补报告能力和分析能力，最后做语义最重的 runtime 扩展。

## Workstream A: Archsim 报告化使用

### 当前判断

`archsim` 已足够用于架构级探索、瓶颈判断、参数扫点和 tradeoff 排序，但它目前主要停留在：

1. `ArchitectureModel + Workload -> behavior/cycle report`
2. `SimModule -> rough ArchitectureModel` 的反推辅助

它目前更适合作为“架构分析报告引擎”，而不是 DSL 的显式上游 IR。

### 目标

明确 `archsim` 的定位：只输出分析报告与 sweep 证据，不定义显式 DSL handoff，不把结构化中间物强耦合进 DSL 流程。

### 计划项

- [x] A1. 明确 archsim 输出物是“报告”而不是 handoff IR
  - 在 README / tutorial 中写清楚：
    1. `BehaviorReport`
    2. `CycleReport`
    3. sweep / ranking evidence
    4. architecture-side PPA recommendations
  - 强调这些结果由 agent / 设计者阅读后再修改 DSL

- [x] A2. 增加 report summary helper
  - 目标：把分散的仿真结果整理成 agent 易读的报告
  - 候选能力：
    1. `summarize_architecture_report(...)`
    2. `emit_architecture_report_markdown(...)`
  - 只输出报告，不产生 DSL 约束文件

- [x] A3. 补一份教程与 worked example
  - 选一个现有设计（优先 `gpu_sm` 或 `sfu`）演示：
    1. 先做 archsim
    2. 生成报告和 sweep 结论
    3. 由 agent 根据报告修改 DSL 结构
    4. 再跑 sim / verify / PPA

### 验收标准

1. 对一个现有案例，能产出稳定、可读的 archsim 报告
2. 报告中至少包含 stage、II、queue、bandwidth、bottleneck、sweep 证据
3. 教程能明确说明“archsim 只给报告，DSL 改动由 agent / 设计者负责”

## Workstream B: 多时钟支持

### 当前判断

当前应明确区分三层能力：

1. legacy DSL / emitted RTL：可以表达多时钟
2. lowered legacy DSL / explicit `SimModule` / `PythonSimulator` / `CompiledSimulator`：
   现在已经可以做显式多时钟 scalar stepping
3. verification collateral / generic batch helpers / generic RTL cosim：
   仍然不是完整的多时钟闭环路径

因此，当前可以宣称“new runtime 已具备显式多时钟执行能力”，但还不能宣称
“所有验证与 collateral 路径都已完成多时钟闭环”。

### 目标

分两阶段推进：

1. 先把支持边界写清楚，并在不支持路径上 fail fast
2. 再为 new runtime 增加真正的多时钟语义支持

### 计划项

- [x] B1. 增加 support matrix 文档
  - 明确列出：
    1. legacy DSL authoring
    2. lowering 到 `SimModule`
    3. PythonSimulator
    4. CompiledSimulator
    5. emitted RTL
    6. local cosim
    7. SV/UVM collateral
  - 对每条路径标注：
    1. 单时钟支持
    2. 多时钟支持
    3. 是否仅 smoke / 是否已 regression lock

- [x] B2. 对 lowering/new runtime 增加 fail-fast guardrail
  - 如果发现多个 clock domain 且目标路径不支持：
    1. 明确报错
    2. 报告 clock/reset 域名
    3. 指出建议路径：legacy sim 或 emitted RTL + external simulator

- [x] B3. 设计 new runtime 的多时钟模型
  - 目标：
    1. 保留 seq block 的 clock/reset domain 元数据
    2. 让 `SimModule` 能表达 domain-aware sequential updates
    3. 为 Python / C++ runtime 提供统一 tick API
  - 候选形式：
    1. `step_clocks({...})`
    2. `tick_domain("wr_clk", ...)`
    3. 或显式时钟边沿事件列表

- [x] B4. 先做 Python runtime 多时钟闭环
  - 覆盖：
    1. 两时钟独立推进
    2. 不同 reset 域
    3. async FIFO 类最小例子

- [x] B5. 再做 compiled simulator 对等支持
  - 保证 Python 与 C++ parity
  - 增加 batch / trace / parity 回归

- [x] B6. 用 `AsyncFIFO` 或最小 CDC 例子做闭环测试
  - 覆盖：
    1. lowered Python sim
    2. compiled sim
    3. emitted RTL cosim（若可行）

### 验收标准

1. 文档中不存在“多时钟好像支持”的模糊表述
2. 当前不支持路径必须 fail fast
3. 若完成 B4/B5，则至少有一个双时钟例子在 Python/C++ runtime 中 regression lock

### 当前进度说明

1. `SimModule` 已加入 `clock_domains` 元数据，`Assignment` / `MemoryWrite`
   可带 `clock_domain`
2. `PythonSimulator` 已支持 `step_clocks(...)`，并有最小双时钟存储/读出
   regression lock
3. `CompiledSimulator` 已支持 `step_clocks(...)`，trace/parity 已覆盖显式双时钟路径
4. legacy multi-clock lowering 已可保留 clock-domain / reset-domain 元数据，
   并对同一时钟下冲突 reset 语义 fail fast
5. verification 侧已区分支持边界：
   1. `run_directed_test` / `run_streaming_*` 对 multi-clock 模块继续明确拒绝
   2. `run_python_uvm_test` 已支持显式 `active_domains` 驱动的 multi-clock
      local closure，可走 Python simulator 与 compiled simulator
   3. generated Python reference model 已支持显式 multi-clock
      `predict_clocks(...)` / smoke
   4. generated SV-UVM collateral 已支持基于 `directed_sequence +
      active_domains` 的显式 multi-clock event-driven 闭环；随机/泛化
      multi-clock UVM 仍未覆盖
6. 仍待后续推进的不是 runtime 多时钟本身，而是：
   1. multi-clock batch API 语义设计
   2. generated SV-UVM collateral 的随机化 / 更通用 multi-clock 闭环
7. generic RTL cosim 已新增显式多时钟 helper：
   `run_legacy_multiclock_rtl_cosim(...)`，可对 legacy multi-clock DUT 做
   lowered Python / compiled / emitted RTL 三方 parity；旧
   `run_legacy_rtl_cosim(...)` 则对 multi-clock DUT 清晰 fail fast

## Workstream C: PPA 面积/功耗

### 当前判断

当前 `ppa` 在模块侧已经能做较好的时序热点定位，也已经能给出一些面积/功耗相关建议，但默认模型仍主要是启发式分数：

1. timing 更偏结构深度和热点表达式
2. area 更偏 state / memory / op count / assignment count
3. power 更偏 state / memory / arithmetic activity proxy

这对排序和找热点是有用的，但对设计者和 agent 来说，还不够“具体”。

### 目标

让面积/功耗分析具备三层报告能力：

1. 启发式结构分析
2. 更具体的热点归因
3. 基于实现报告的校准估计

### 计划项

- [x] C1. 增强面积/功耗热点归因
  - 增加更清晰的 evidence 输出，例如：
    1. 最大 memory 宽度/深度/bit 数
    2. 最重 multiplier / adder / mux 热点
    3. 大状态块和高切换代理对象
    4. 模块名、目标信号、源码行号

- [x] C2. 增加模块侧 area/power breakdown
  - 将当前总分拆成可解释子项：
    1. state contribution
    2. memory contribution
    3. arithmetic contribution
    4. mux/control contribution
    5. write activity contribution

- [x] C3. 增加 architecture-side area/power proxy
  - 在 archsim 侧先不追求物理精度
  - 先补：
    1. stage activity proxy
    2. bytes moved proxy
    3. queue occupancy proxy
    4. compute utilization proxy
  - 用于早期比较“更宽 / 更深 / 更多 bank / 更多 FU”的相对代价

- [x] C4. 打通 calibration workflow
  - 强化：
    1. timing/area/power report 解析
    2. calibration sample 构建
    3. calibration model 拟合
    4. calibrated estimate 输出
  - 目标是让用户知道何时应信 heuristic，何时应信 calibrated estimate

- [x] C5. 给 agent 更具体的可执行建议
  - 例如：
    1. 指明模块名和源码行号
    2. 指明哪类资源是面积主因
    3. 指明哪类路径可能是功耗主因
    4. 对乘法器类热点给出“拆分 / 共享 / 分级流水 / LUT 近似”的分析建议
  - 明确这些建议只出现在报告中，不自动传给 DSL，也不自动改写 RTL

- [x] C6. 补教程与示例
  - 更新 README / `TUTORIAL_ARCH_PPA.md`
  - 说明：
    1. heuristic score 的意义
    2. calibrated estimate 的意义
    3. area/power 该如何解读与使用

### 验收标准

1. PPA report 能输出面积/功耗 breakdown，而不是只有总建议
2. 至少一个案例能显示校准前后结果差异
3. README / tutorial 中明确说明 heuristic 与 calibrated estimate 的边界

## 完成判据

当以下条件全部满足时，本计划可视为完成：

1. 用户和 agent 都能清楚知道当前单时钟/多时钟支持矩阵
2. 架构仿真结果可以通过稳定报告支撑 DSL 设计决策
3. PPA 对 timing/area/power 三者都能给出可解释证据
4. 多时钟若尚未完全支持，也必须被 guardrail 明确挡住
5. 若多时钟 runtime 已做完，至少有一个双时钟设计被回归锁定
