# RTLCraft Spec-to-DSL 细化生成修改计划

日期：2026-06-10

## 1. 审阅结论

当前项目的核心问题不是 Python DSL AST、Simulator、VerilogEmitter 或 UVM 后端能力不足，而是：

```text
顶层 spec
  -> Architecture / Microarchitecture IR
  -> 可执行 DSL 生成
```

这段中间流水线没有真正闭合。项目能生成 markdown spec、skeleton、部分 reference 信息，但实际运行时经常退化为“保留骨架”，没有生成细致的底层 Python DSL 代码，也没有把架构/微架构细化成可执行 RTL 结构。

一次只读验证命令显示：

```text
python3 -m rtlgen.skill_ppa --skill hetero_riscv4 --stage skeleton spec_gen dsl_from_spec --output /private/tmp/rtlc_audit_out

dsl_from_spec_success: 0
dsl_from_spec_fallback: 14
```

也就是说，14 个模块全部 fallback 到 skeleton，没有任何一个模块由 spec 自动生成 DSL。

## 2. 关键问题

### 2.1 `DSLGenerator` API 断裂

`rtlgen/decomposition.py` 中 `generate_dsl_skeleton()` 试图导入：

```python
from rtlgen import ArchitecturePlanner, DSLGenerator, SpecCompleter
```

但 `rtlgen.__init__` 没有导出 `DSLGenerator`，实际调用会失败。仓库中也没有 README 所描述的 `rtlgen/dsl_gen.py`，实际存在的是历史用途的 `rtlgen/dsl_generator.py`，主要用于从 markdown spec 通过少数硬编码 PE 类型拼接 DSL 代码。

### 2.2 `ArchitecturePlanner` 操作粒度过粗

`rtlgen/arch_planner.py::_extract_ops()` 最后把表达式压成一条粗粒度 assign：

```text
y = a * b + c -> ["assign y = a * b + c"]
```

因此即使 `latency_max = 3`，stage 也会变成：

```text
stage0: assign y = a * b + c
stage1: empty
stage2: empty
```

这无法指导乘法、加法、寄存器切分、valid/ready、旁路、stall 等微架构生成。

### 2.3 `dsl_from_spec` 不是生成器

`rtlgen/skill_ppa.py::_run_dsl_from_spec()` 当前逻辑是：

1. 如果有手写 `{pe_name}.py`，尝试加载；
2. 否则保留 skeleton；
3. 即使全部 fallback，也标记 PASS。

这会掩盖“没有生成 DSL”的真实问题。

### 2.4 层次化设计被 prompt 压平成单体模块

`rtlgen/prompt_builder.py` 中明确要求：

```text
Do NOT instantiate sub-modules as separate classes.
Implement ALL steps as a SINGLE monolithic module.
```

这和项目希望支持模块分解、层次化微架构、子模块互联的目标冲突。

### 2.5 子模块端口规格不规范

生成的子模块 spec 中会出现：

```text
addrgen_btb_index[9:0]
```

被当作端口名，而不是：

```text
name = addrgen_btb_index
width = 10
```

并且宽度可能被推成 64。这会污染后续 DSL 生成输入。

## 3. 修改目标

目标不是一次性让所有 CPU/GPGPU/NoC 复杂模块完美生成，而是先建立一个真实、可测试、不可假成功的闭环：

```text
SpecIR
  -> ArchitectureIR / MicroArchitectureIR
  -> DSLGenerator
  -> Module AST
  -> Simulator / VerilogEmitter / Lint
```

同时保留 skill/reference/agent 体系，让它们用于补充复杂逻辑，而不是替代基础生成器。

## 4. 修改计划

### P0：修复 API 断裂和假成功

文件范围：

- `rtlgen/__init__.py`
- `rtlgen/decomposition.py`
- `rtlgen/skill_ppa.py`
- 新增或恢复 `rtlgen/dsl_gen.py`
- tests

计划：

1. 新增真正的 `DSLGenerator(spec: SpecIR, arch: ArchitectureIR)`。
2. 在 `rtlgen.__init__` 导出 `DSLGenerator`。
3. 修复 `generate_dsl_skeleton()` 的导入和调用。
4. 修改 `_run_dsl_from_spec()`，没有生成 DSL 时不能静默 PASS。
5. 增加最小回归测试：
   - `BehavioralSpec -> SpecIR -> ArchitecturePlanner -> DSLGenerator`
   - `SpecIR MAC -> DSL Module -> VerilogEmitter`
   - `dsl_from_spec` 全 fallback 时必须报告失败或 `needs_generation`。

验收标准：

```text
from rtlgen import DSLGenerator
```

可用，并且一个简单 MAC spec 能生成可实例化、可仿真、可 emit Verilog 的 DSL Module。

### P1：细化 ArchitectureIR / 增加 MicroArchitectureIR

文件范围：

- `rtlgen/spec_ir.py`
- `rtlgen/arch_planner.py`
- tests

计划：

1. 增加结构化 IR：
   - `OperationSpec`
   - `PipelineStageSpec`
   - `RegisterTransferSpec`
   - `FSMStateSpec`
   - `HandshakeSpec`
   - `SubmoduleInstanceSpec`
   - `ConnectionSpec`
2. 用 Python `ast` 或专用表达式解析器替换字符串扫描。
3. 将 `y = a * b + c` 分解为：

```text
op0: tmp_mul = a * b
op1: y = tmp_mul + c
```

4. 根据 PPA/timing 分配 stage 和 operator implementation。
5. 支持 `function.operations` 的 ALU opcode 生成。

验收标准：

```text
MAC latency_max=3
```

能产生非空 stage 列表、明确中间信号、明确寄存器边界。

### P2：实现确定性的 DSL 生成器

文件范围：

- `rtlgen/dsl_gen.py`
- `rtlgen/core.py` 如需补充 helper
- `rtlgen/logic.py` 如需补充 helper
- tests

计划：

1. `comb_alu` 生成：
   - ports
   - wires
   - Switch/opcode
   - default assignment
   - width handling
2. `stream_pipeline` 生成：
   - stage regs
   - valid pipeline
   - ready/stall
   - optional skid buffer
   - reset behavior
3. `fsm_controller` 生成：
   - state register
   - next-state logic
   - output decode
4. `hierarchical` 生成：
   - child Module skeleton
   - `SubmoduleInst`
   - internal wires
   - explicit port map
5. 端口解析规范化：
   - `foo[9:0] -> name=foo, width=10`
   - `foo[31] -> name=foo, width=1 or bit-select context`

验收标准：

生成出的 DSL 不只是 ports + pass，而是至少包含完整 comb/seq AST。

### P3：让 skill/reference/agent 成为细化生成输入

文件范围：

- `rtlgen/gen_requirement.py`
- `rtlgen/reference_extractor.py`
- `rtlgen/prompt_builder.py`
- `rtlgen/spec_markdown.py`
- `rtlgen/skill_ppa.py`
- 可能新增 `rtlgen/agent_dsl_generator.py`

计划：

1. `spec_gen` 同时输出机器可读 `GenerationContext` JSON/YAML，不只输出 markdown。
2. `ReferenceExtractor` 提取：
   - 端口模式
   - 状态变量
   - FSM 摘要
   - handshake 模式
   - 关键 comb/seq 片段
3. `prompt_builder` 增加 `hierarchy_mode`：
   - `monolithic`
   - `hierarchical`
   - `leaf_only`
4. 去掉固定“不要实例化子模块”的硬编码要求。
5. 增加 `AgentDSLGenerator`：
   - 先本地确定性生成 skeleton + common logic；
   - 再让 agent/LLM 补复杂行为；
   - parse/sim 失败时进入 repair。

验收标准：

同一个 PE spec 能在无手写 `{pe_name}.py` 的情况下生成至少一版 DSL，并进入解析和仿真检查，而不是直接 fallback。

### P4：行为级到 DSL 的验证闭环

文件范围：

- `rtlgen/behavior_roundtrip.py`
- `rtlgen/dsl_sim.py`
- `rtlgen/verifier.py`
- `rtlgen/skill_ppa.py`
- tests

计划：

1. `dsl_from_spec` 之后自动执行：
   - DSL parse/import
   - Module instantiate
   - reset simulation
   - golden vector simulation
   - behavior roundtrip
   - lint
2. 失败时输出结构化错误：
   - parse error
   - missing port
   - wrong width
   - sim mismatch
   - unconnected output
3. 只有生成、解析、仿真至少通过最小门槛，stage 才能 PASS。

验收标准：

pipeline 输出中必须能区分：

```text
generated
parsed
sim_passed
roundtrip_passed
fallback
needs_agent
failed
```

### P5：文档和 README 对齐

文件范围：

- `README.md`
- `README_CN.md`
- `RTLGEN.md`
- `Tutorial.md`

计划：

1. 修正文档中的 `dsl_gen.py` / `dsl_generator.py` 命名漂移。
2. 明确哪些是已实现、哪些是 roadmap。
3. 增加一个最小可运行例子：

```text
MAC16 YAML spec
  -> ArchitecturePlanner
  -> DSLGenerator
  -> Simulator

```

验收标准：

README 中的主流程命令能在当前仓库直接运行。

## 5. 推荐执行顺序

建议先做一个窄而完整的 vertical slice：

```text
P0 + P1 + P2 的 stream_pipeline MAC
```

完成后再扩展：

```text
ALU opcode
  -> FSM controller
  -> ready-valid pipeline
  -> hierarchical submodule
  -> skill_ppa 集成
```

这样可以最快证明项目真正具备：

```text
顶层 spec -> 细化架构/微架构 -> Python DSL -> 仿真/Verilog
```

的闭环能力。

## 6. 下一步建议

如果确认此计划，建议第一轮修改只做 P0 和一个最小 P1/P2 示例，避免一开始就进入 CPU/GPGPU 复杂模块。第一轮目标是让 `DSLGenerator` 成为真实可用的公共 API，并让 `dsl_from_spec` 不再用 fallback 掩盖失败。

## 7. 分层 IR、仿真验证与智能体 Lowering 计划

### 7.1 总原则

后续框架应遵循一个核心原则：

```text
Agent-driven lowering, tool-enforced refinement.
```

中文表述：

```text
智能体驱动降层，工具链强制验证细化关系。
```

也就是说，智能体可以负责把上一层 IR 转换、细化、补全为下一层 IR 或 DSL 实现；但智能体不能自己宣称成功。每次 lowering 后，必须由确定性工具链完成：

```text
schema validate
type/width check
protocol check
simulation
trace comparison
coverage/lint/PPA check
```

只有验证通过，下一层 IR 才能被接受。

### 7.2 每层 IR 的基本要求

每一层 IR 都必须满足：

1. **可序列化**：能导出 JSON/YAML/dict，便于存档、diff、agent 输入和用户审阅。
2. **可 review**：能生成面向用户的 markdown/table/diagram。
3. **可仿真或解释执行**：即使不是 bit-level RTL，也要能产生可比较的 observation。
4. **可与前一级对比**：定义明确的 refinement relation。
5. **可下推**：能作为下一层 lowering 的输入，而不是只停留在文档。

建议统一 observation 格式：

```python
Observation = {
    "cycle": int | None,
    "inputs": {...},
    "outputs": {...},
    "state_events": [...],
    "transactions": [...],
    "tracepoints": {...},
}
```

不同层不一定比较全部内部细节，而是比较经过 projection 后的可观察行为。

### 7.3 建议的 IR 分层

#### L0: SpecIR

职责：

- 用户需求
- 功能描述
- 端口/协议
- timing/PPA 目标
- reset/clock 假设
- verification 目标

验证方式：

- schema check
- 端口完整性检查
- 约束一致性检查
- 自动生成 directed/random tests 的 seed 集

与下一层对比：

- `BehaviorIR` 必须覆盖 `SpecIR` 中所有 required behavior、ports、protocol 和 PPA/timing assumptions。

#### L1: BehaviorIR

职责：

- 无时序或弱时序的功能行为
- 输入到输出的函数关系
- 边界条件
- 异常路径
- 高层状态含义

当前问题：

- 现在很多行为是 `Callable[[CycleContext], None]`，能跑但不易序列化、不易 review。

建议新增结构：

```python
ExprIR
BehaviorRuleIR
OutputRuleIR
AssumptionIR
InvariantIR
```

验证方式：

- functional interpreter
- 与 `SpecIR` 自动生成测试对比
- invariant check

用户 review：

- 行为规则表
- 输入/输出关系表
- 边界条件表
- 异常/默认行为列表

#### L2: CycleIR

职责：

- 周期级行为
- state/register
- next-state equation
- reset
- enable
- valid/ready
- latency
- stall/flush
- trace event

建议新增结构：

```python
StateIR
StateUpdateIR
CycleRuleIR
HandshakeRuleIR
TraceEventIR
```

验证方式：

- cycle-level interpreter
- transaction 对齐
- latency-aware comparison

与前一级对比：

```text
BehaviorIR(input transaction) == CycleIR(output transaction after latency)
```

用户 review：

- 寄存器表
- next-state 表
- reset 表
- valid/ready 时序表
- latency 表

#### L3: MicroArchIR

职责：

- pipeline stage
- FSM
- op dependency graph
- resource binding
- memory/cache structure
- bypass
- hazard/stall/flush
- arbitration
- buffer/FIFO

建议新增结构：

```python
OperationIR
PipelineStageIR
PipelineRegisterIR
FSMIR
MemoryIR
ResourceBindingIR
HazardRuleIR
ArbiterIR
BufferIR
```

验证方式：

- microarchitecture schedule simulator
- pipeline trace simulator
- FSM transition checker
- protocol checker
- PPA pre-check

与前一级对比：

```text
CycleIR observable trace == MicroArchIR projected trace
```

允许 MicroArchIR 拥有更多内部状态，但外部 transaction、latency、tracepoint 必须满足 CycleIR。

用户 review：

- pipeline diagram
- stage operation table
- FSM 状态转移表
- memory/cache 结构表
- hazard/bypass 规则表
- PPA 预估表

#### L4: StructuralIR

职责：

- module hierarchy
- submodule instance
- interface bundle
- port map
- internal wire
- protocol adapter
- clock/reset domain

建议新增结构：

```python
ModuleIR
InstanceIR
PortIR
InterfaceIR
ConnectionIR
ClockDomainIR
ResetIR
AdapterIR
```

验证方式：

- port/interface completeness check
- width check
- clock/reset check
- no dangling wire
- no multiple driver
- protocol compatibility check

与前一级对比：

```text
MicroArchIR sub-blocks/stages/resources
  refined by
StructuralIR modules/instances/connections
```

用户 review：

- 子模块实例表
- port map 表
- interface connection 表
- clock/reset domain 表
- block diagram

#### L5: RTLIR / DSL AST

职责：

- 当前 `Module/Input/Output/Wire/Reg/Array/If/Switch/Assign/SubmoduleInst`
- 可 emit Verilog
- 可 RTL-level simulation
- 可 lint/PPA/SMT/cosim

验证方式：

- 现有 `Simulator`
- VerilogEmitter
- VerilogLinter
- PPAAnalyzer
- optional iverilog/cosim
- optional SMT

与前一级对比：

```text
StructuralIR interface/connection/tracepoint
  refined by
DSL AST concrete signals and assignments
```

用户 review：

- DSL source
- generated Verilog
- source map
- lint/PPA report
- waveform/trace report

#### L6: VerificationIR

职责：

- golden tests
- random tests
- coverage bins
- assertions
- tracepoints
- UVM/cocotb generation intent
- scoreboard mapping

当前问题：

- verification 信息分散在 `verif_gen.py`、`uvmgen.py`、`behavior_roundtrip.py`、`skill_ppa.py`。

建议统一为：

```python
TestVectorIR
CoverageBinIR
AssertionIR
TracepointIR
ScoreboardIR
VerificationPlanIR
```

验证方式：

- 每层 IR lowering 后都绑定同一份 VerificationIR 的对应 projection。

用户 review：

- 测试向量表
- 覆盖率目标表
- assertion 列表
- scoreboard 对比规则

### 7.4 智能体在各层 Lowering 中的职责

智能体适合做有歧义、有工程经验依赖、需要参考样例的 lowering 工作。

#### SpecIR -> BehaviorIR

智能体职责：

- 补全未明确的边界条件
- 明确默认行为
- 明确异常路径
- 把自然语言需求变成结构化行为规则

工具链检查：

- 行为是否覆盖所有输出
- 是否存在未定义输入组合
- 是否违反 SpecIR assumptions

#### BehaviorIR -> CycleIR

智能体职责：

- 判断哪些变量需要变成状态
- 决定 latency
- 添加 valid/ready
- 添加 reset/enable
- 定义 trace event

工具链检查：

- cycle simulation 是否与 BehaviorIR transaction 对齐
- reset 是否完整
- state update 是否单写
- latency 是否满足 timing

#### CycleIR -> MicroArchIR

智能体职责：

- 切 pipeline stage
- 选择 FSM
- 添加 stall/flush
- 设计 bypass/hazard
- 选择资源绑定
- 选择 FIFO/cache/arbiter 等微架构组件

工具链检查：

- pipeline trace 是否满足 CycleIR
- FSM 是否无死锁
- valid/ready 是否无丢包/重复
- PPA 是否初步满足目标

#### MicroArchIR -> StructuralIR

智能体职责：

- 拆子模块
- 定义实例化层次
- 建立接口和 port map
- 加协议 adapter

工具链检查：

- 所有端口已连接
- 位宽匹配
- clock/reset domain 合法
- 无多驱动/悬空

#### StructuralIR -> DSL AST

智能体职责：

- 写 Python DSL
- 补 comb/seq 逻辑
- 连接子模块
- 根据仿真/PPA/lint 反馈修复

工具链检查：

- DSL parse/import
- Simulator
- roundtrip comparison
- lint
- PPA
- Verilog emission

### 7.5 Lowering 接受流程

每次 lowering 都采用同一流程：

```text
IR_i
  -> agent proposes IR_{i+1}
  -> schema validation
  -> static validation
  -> simulator/interpreter run
  -> compare observations with IR_i
  -> generate structured report
  -> accept or repair
```

失败报告应结构化，例如：

```json
{
  "layer": "CycleIR",
  "previous_layer": "BehaviorIR",
  "status": "mismatch",
  "test": "load_miss_refill",
  "cycle": 7,
  "signal": "valid_out",
  "expected": 1,
  "actual": 0,
  "likely_cause": "REFILL state did not transition to DELIVER"
}
```

智能体拿到该报告后只能做局部 patch，并再次进入验证流程。

### 7.6 用户 Review 产物

每一轮生成应产出 review bundle：

```text
generated/<skill>/review/
  01_spec_review.md
  02_behavior_review.md
  03_cycle_review.md
  04_microarch_review.md
  05_structure_review.md
  06_verification_plan.md
  07_lowering_report.md
```

其中重点：

- `03_cycle_review.md`：寄存器、next-state、latency、valid/ready。
- `04_microarch_review.md`：pipeline、FSM、memory/cache、hazard/bypass、PPA。
- `05_structure_review.md`：子模块实例、接口、连接、clock/reset。
- `07_lowering_report.md`：每层 lowering 是否通过，失败原因和修复次数。

### 7.7 第一阶段建议实现范围

第一阶段不要试图覆盖所有复杂设计。建议先实现：

```text
SpecIR
  -> BehaviorIR
  -> CycleIR
  -> MicroArchIR
  -> StructuralIR
  -> DSL AST
```

的最小闭环，目标模块：

```text
stream_pipeline MAC
```

最小验收：

1. 每层 IR 可导出 JSON。
2. 每层可生成 markdown review。
3. 每层有 simulator/interpreter 或 checker。
4. 每层能与前一级比较。
5. DSL AST 能仿真并 emit Verilog。
6. `dsl_from_spec` 不允许全 fallback 仍 PASS。

完成这个 vertical slice 后，再扩展到：

```text
opcode ALU
FSM controller
ready-valid FIFO pipeline
cache refill FSM
NoC router
hierarchical CPU block
```
