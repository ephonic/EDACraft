## 1. Product Thesis

RTLCraft should stop behaving like a workflow framework and become a compact
toolbox of hard engineering capabilities.

The agent is the orchestrator.

The repo should provide strong tools, not a heavy process:

1. lightweight architecture exploration
2. reliable executable design modeling where it creates real leverage
3. fast, trustworthy compiled simulation
4. verification generation rooted in executable behavior
5. PPA-oriented analysis and recommendations

Everything else is optional, legacy, or removable.

## 2. Design Principles

### 2.1 Capability-first

Each subsystem should be valuable on its own:

1. `archsim` should help with architecture tradeoffs
2. compiled simulation should help validate detailed behavior
3. verification generation should help produce checkers and UVM collateral
4. PPA tooling should help improve designs

If a module only makes the process look more structured, it is probably not core.

### 2.2 No repo-wide contract layer

We should not build a central `contract.py` or a mandatory shared object model for
all tools.

Instead:

1. each capability owns the smallest local model it needs
2. adapters are allowed only when there is clear payoff
3. integration should happen through agent orchestration, not framework coupling

This keeps the system easier to evolve and avoids over-constraining the agent.

### 2.3 No forced tool coupling

Architecture simulation, compiled simulation, verification, and PPA should be able
to evolve independently.

They may interoperate, but they should not be forced into one shared internal API.

### 2.4 Compiled hot path

For detailed simulation, Python is an interface layer only.

The hot execution path should live in compiled C++:

1. better speed
2. better determinism
3. a more trustworthy base for verification and optimization

## 3. Capability Scope

## 3.1 Capability A — Architecture Simulator

### Goal

Provide a lightweight but general architecture simulator for early reasoning.

### Required coverage

1. CPU pipelines and memory systems
2. GPGPU throughput machines and memory hierarchies
3. NPU / accelerator dataflows and buffering
4. control-oriented blocks and protocol controllers
5. generic datapaths, queues, arbiters, and streaming pipelines

### Required simulation levels

1. behavior-level mode
   - very fast
   - good for throughput, bandwidth, contention, and topology exploration
2. cycle-level mode
   - lightweight timing-aware execution
   - good for backpressure, occupancy, arbitration, and queueing analysis

### Model philosophy

This simulator should own a very small local modeling API.

It does not need to share a repo-wide IR with other subsystems.

## 3.2 Capability B — Executable Design Model

### Goal

Keep an executable design model only where it materially improves the simulator /
verification / PPA stack.

### Required properties

1. precise semantics
2. good agent editability
3. explicit width, signedness, reset, and clock behavior
4. straightforward lowering to RTL when needed

### Constraint

This is not a document layer and not a framework-wide contract.

It is simply the local execution model for the detailed simulator stack.

## 3.3 Capability C — Compiled Simulator

### Goal

Make detailed simulation fast enough and trustworthy enough that an agent prefers
to use it by default.

### Required features

1. correct arithmetic and signedness behavior
2. deterministic reset/step semantics
3. stable tracing and replay
4. compiled C++ execution
5. differential checking against emitted RTL when available

### Architecture

```text
agent / Python API
  -> simulator-facing executable model
  -> normalized simulation IR
  -> C++ code generation / compilation
  -> compiled runtime
  -> trace / debug / differential-check hooks
```

### Reliability bar

1. golden tests for semantic behavior
2. regression coverage for arithmetic, reset, FSM, memory, slicing, concat
3. parity checks between Python-side expectations and C++ execution
4. differential cosim against RTL where practical

If the simulator is wrong, the whole reboot fails.

## 3.4 Capability D — Verification Bridge

### Goal

Generate reusable verification artifacts from executable behavior.

### Target outputs

1. reference models
2. scoreboards
3. directed tests
4. transaction-level checkers
5. UVM skeletons, monitors, sequences, and environments

### Principle

Verification should be derived from runnable behavior, not from markdown template
completeness.

## 3.5 Capability E — PPA Advisor

### Goal

Help the agent improve the design rather than only describe it.

### Required outputs

1. structural recommendations
2. latency / area / power tradeoff suggestions
3. transform candidates
4. quick what-if exploration support
5. optional feedback from synthesis, lint, and simulation traces

## 4. What Is No Longer Core

The following should stop defining product identity:

1. document-driven flow orchestration
2. multi-layer authored IR chains
3. approval-gate mechanics
4. review bundles as the main user surface
5. feedback plumbing whose purpose is process choreography

These may remain temporarily as compatibility code, but they should not drive the
new architecture.

## 5. Repo Restructure Direction

The reboot should happen in a clean area under `rtlgen_x/`.

### Core capability areas

1. `rtlgen_x/archsim/`
2. `rtlgen_x/sim/`
3. future `rtlgen_x/verify/`
4. future `rtlgen_x/ppa/`

### Rules for this clean area

1. no shared `contract.py`
2. no central control-plane logic
3. no mandatory document machinery
4. no coupling layer that forces unrelated tools into one schema

## 6. Execution Order

## Phase 1 — Establish the clean-core baseline in `rtlgen_x/`

1. create a clean package boundary
2. add the first architecture simulator primitives
3. add a self-contained compiled-simulator model and scaffold
4. regression-lock the baseline with focused tests

## Phase 2 — Harden the compiled simulator

1. strengthen semantics for arithmetic, reset, memory, and state updates
2. lower to normalized simulation IR internally
3. emit and compile C++ kernels
4. add parity and differential tests
5. measure performance

## Phase 3 — Expand architecture simulation

1. enrich behavior-level modeling
2. enrich cycle-level timing primitives
3. add reusable queue / scheduler / memory / interconnect components
4. validate on CPU, GPGPU, NPU, controller, and datapath examples

## Phase 4 — Build verification capability

1. derive checkers and reference models from executable behavior
2. generate directed tests
3. generate UVM skeleton collateral

## Phase 5 — Build PPA capability

1. define cost models
2. analyze structural opportunities
3. rank optimization suggestions
4. connect trace-derived evidence to PPA recommendations

## 7. Immediate Implementation Guidance

From this point on:

1. prefer local capability models over shared contracts
2. prefer direct capability APIs over orchestration layers
3. keep new code in `rtlgen_x/` unless compatibility work requires otherwise
4. only preserve old flow machinery when needed for migration or reference

## 8. Acceptance Criteria

This reboot is successful if:

1. `rtlgen_x/` clearly centers on hard capabilities
2. the architecture simulator is useful for real tradeoff exploration
3. the compiled simulator is both fast and semantically trustworthy
4. verification generation is rooted in executable behavior
5. PPA guidance becomes an active capability
6. agents can use each capability directly, without going through a workflow framework

## 9. Bottom Line

The new direction is simple:

```text
No control plane.
No mandatory document flow.
No repo-wide contract.
No forced coupling.

Just strong tools.
```
