# Cross-Layer Constraint & Intent Framework — Implementation Plan

**Project**: RTLCraft / Smart Earphone SoC  
**Date**: 2026-06-14  
**Version**: 0.1  
**Status**: Proposal / Ready for review

---

## 1. Background & Motivation

The current Spec2RTL flow defines a 6-layer IR:

```
SpecIR → BehaviorIR → CycleIR → ArchitectureIR → StructuralIR → DSL/Verilog
```

Each layer refines the design from requirements to implementation. However, **constraints and verification intent** (functional, performance, power, timing) are currently scattered across:

- Markdown documents (`design_spec.md`, `README.md`)
- Hard-coded Python tests (`run_functional_tests`, `run_dsl_sim_tests`)
- Ad-hoc comments in RTL
- Post-hoc lint / synthesis reports

More importantly, the flow is currently **unidirectional**: once a decision is made at a high layer, there is no structured mechanism for lower-layer implementation details to feed back upward when they reveal that a constraint is infeasible.

This plan proposes a **unified, generic Cross-Layer Constraint & Intent Framework** that:

1. Builds on the existing `rtlgen/contracts.py` layer/contract infrastructure.
2. Makes constraints first-class IR artifacts attached to IR entities.
3. Propagates constraints **downward** across layers via semantic transforms.
4. Validates constraints **upward** from implementation back to specification, surfacing conflicts as structured feedback.
5. Standardizes the agent's design workflow using a reusable scaffold of layers, gates, and decision records.

This turns the design process into an iterative, auditable loop rather than a one-way pipeline.

---

## 2. Goals

1. **Genericity**: Framework is not Earphone-specific; reusable across RTLCraft projects.
2. **Bidirectional cross-layer propagation**: Constraints flow **down** (refinement) and validation results flow **up** (feedback/rollback).
3. **Traceability**: Every generated artifact can be traced back to its originating requirement, and every requirement conflict can be traced to its blocking implementation detail.
4. **Human-AI collaboration clarity**: Explicitly distinguish human-defined constraints from AI-generated implementations and feedback.
5. **Agent workflow standardization**: Provide a scaffold (layers → gates → decisions → feedback) that all RTLCraft agents must follow.
6. **Minimal intrusion**: Extend existing RTLGen core without breaking current designs.
7. **Actionable output**: Generate UVM/cocotb testbenches, SVA assertions, constraint traceability reports, and design-issue reports.

## 3. Core Abstractions

### 3.1 `IRConstraint` — The First-Class Constraint Object

```python
from dataclasses import dataclass, field
from typing import Any, Callable, Dict, List, Optional, Tuple

@dataclass(frozen=True)
class IRConstraint:
    uid: str
    name: str
    category: str          # "functional" | "performance" | "power" | "timing" | "verification"
    layer: str             # "SpecIR" | "BehaviorIR" | "CycleIR" | "ArchitectureIR" | "StructuralIR" | "DSL" | "Verilog"
    expr: Any              # Layer-specific expression: Python predicate / SVA / timing budget / coverage point
    target: Optional[str]  # module name / signal name / port path
    unit: Optional[str]    # mW/MHz | ns | cycles | %
    owner: str = "human"   # "human" | "ai" — distinguishes human-defined from AI-derived
    source_ref: Optional[str] = None   # e.g., "design_spec.md#RV32M"
    derived_from: Tuple[str, ...] = () # upstream constraint uids
    metadata: Dict[str, Any] = field(default_factory=dict)
```

Specialized subtypes:

```python
class FunctionalConstraint(IRConstraint): ...
class PerformanceConstraint(IRConstraint): ...
class PowerConstraint(IRConstraint): ...
class TimingConstraint(IRConstraint): ...
class VerificationIntent(IRConstraint): ...  # kind="sequence" | "scoreboard" | "coverage" | "assertion"
```

### 3.2 `IREntity` — Attachment Point

Constraints can be attached to any IR node, not just modules.

```python
class IREntity:
    def __init__(self, name: str):
        self.name = name
        self.constraints: List[IRConstraint] = []
        self._layer: str = "StructuralIR"

    def add_constraint(self, c: IRConstraint) -> "IREntity":
        self.constraints.append(c)
        return self

    def constraints_by(self, *, category=None, layer=None, owner=None) -> List[IRConstraint]:
        result = self.constraints
        if category:
            result = [c for c in result if c.category == category]
        if layer:
            result = [c for c in result if c.layer == layer]
        if owner:
            result = [c for c in result if c.owner == owner]
        return result
```

`Module`, `Signal`, `Input`, `Output`, `Wire`, `Reg`, `StateDesc`, and `SubmoduleInst` should all support this interface.

### 3.3 `ConstraintPropagator` — Bidirectional Transformation Engine

Constraints are not simply copied; they are **semantically transformed** between layers. The engine supports both forward (refinement) and backward (feedback) propagation.

```python
ForwardTransform = Callable[[IRConstraint, str, str], Optional[IRConstraint]]
BackwardTransform = Callable[[IRConstraint, str, str], Optional["ConstraintFeedback"]]

class ConstraintPropagator:
    def __init__(self):
        self._forward_rules: Dict[Tuple[str, str], List[ForwardTransform]] = {}
        self._backward_rules: Dict[Tuple[str, str], List[BackwardTransform]] = {}

    def register_forward(self, src_layer: str, dst_layer: str, transform: ForwardTransform):
        self._forward_rules.setdefault((src_layer, dst_layer), []).append(transform)

    def register_backward(self, src_layer: str, dst_layer: str, transform: BackwardTransform):
        """Validate constraints from dst_layer against src_layer expectations."""
        self._backward_rules.setdefault((src_layer, dst_layer), []).append(transform)

    def propagate_forward(self, constraints: List[IRConstraint], src: str, dst: str) -> List[IRConstraint]:
        out = []
        for c in constraints:
            for t in self._forward_rules.get((src, dst), []):
                nc = t(c, src, dst)
                if nc is not None:
                    out.append(nc)
        return out

    def propagate_backward(self, constraints: List[IRConstraint], src: str, dst: str) -> List["ConstraintFeedback"]:
        """Check whether dst-layer constraints can satisfy src-layer constraints."""
        out = []
        for c in constraints:
            for t in self._backward_rules.get((src, dst), []):
                fb = t(c, src, dst)
                if fb is not None:
                    out.append(fb)
        return out

    def propagate_all(self, entity: IREntity, path: List[Tuple[str, str]]) -> List[IRConstraint]:
        constraints = list(entity.constraints)
        for src, dst in path:
            constraints = self.propagate_forward(constraints, src, dst)
        return constraints
```

### 3.4 `ConstraintFeedback` — Upstream Issue Reporting

When a lower-layer implementation cannot meet an upper-layer constraint, the framework emits structured feedback instead of silently failing.

```python
from enum import Enum

class FeedbackSeverity(Enum):
    INFO = "info"
    WARNING = "warning"
    VIOLATION = "violation"
    BLOCKER = "blocker"

@dataclass(frozen=True)
class ConstraintFeedback:
    uid: str
    severity: FeedbackSeverity
    source_constraint_uid: str   # the upper-layer constraint that is impacted
    detected_at_layer: str       # e.g., "Verilog" or "StructuralIR"
    message: str
    suggested_resolutions: List[str]  # e.g., ["relax SpecIR power budget", "switch to faster divider architecture"]
    owner: str = "ai"            # who detected the issue
```

## 5. Cross-Layer Constraint Propagation Examples

### 5.1 Example A: Functional Constraint — RV32M Division by Zero

| Layer | Constraint | Owner | Derived From |
|-------|-----------|-------|--------------|
| **SpecIR** | `DIV/REM by zero -> -1; DIVU/REMU by zero -> MAX/dvd` | human | — |
| **BehaviorIR** | ISS golden test: `iss_div_zero_test()` | ai | SpecIR |
| **CycleIR** | Divider busy for 32 cycles when divisor == 0 | ai | SpecIR |
| **ArchitectureIR** | `div_divisor==0 && div_done -> div_result==0xFFFFFFFF` | ai | CycleIR |
| **StructuralIR** | Monitor `div_result` in execute-stage agent | ai | ArchitectureIR |
| **Verilog** | UVM sequence + SVA assertion | ai | StructuralIR |

### 5.2 Example B: Performance/Power Constraint — CPU Active Power

| Layer | Constraint | Owner | Derived From |
|-------|-----------|-------|--------------|
| **SpecIR** | CPU active power < 0.5 mW/MHz | human | — |
| **BehaviorIR** | Prefer iterative algorithms over combinational | ai | SpecIR |
| **CycleIR** | Stall pipeline during multi-cycle ops | ai | SpecIR |
| **ArchitectureIR** | `core_clk_en = ~core_stall & ~muldiv_busy` | ai | CycleIR |
| **StructuralIR** | Insert ICG before pipeline register groups | ai | ArchitectureIR |
| **Verilog** | `if (clk_en) reg <= next` coding style + power report | ai | StructuralIR |

### 5.3 Example C: Bottom-Up Feedback — Power Budget Violation

This example shows the **feedback loop**: when detailed implementation reveals that a SpecIR constraint cannot be met.

| Direction | Layer | Finding | Feedback Severity | Suggested Resolution |
|-----------|-------|---------|-------------------|---------------------|
| Forward | SpecIR | CPU active power < 0.5 mW/MHz | — | — |
| Forward | ArchitectureIR | `core_clk_en = ~core_stall & ~muldiv_busy` | — | — |
| Forward | StructuralIR | ICG inserted on 12 register groups | — | — |
| **Backward** | **Verilog / Synthesis** | **Estimated dynamic power = 0.72 mW/MHz at 160 MHz, 22nm** | **VIOLATION** | **Relax SpecIR budget to 0.8 mW/MHz, or add power domain, or reduce pipeline depth** |
| Backward | StructuralIR | Power still 0.65 mW/MHz after additional clock gating | WARNING | Add operand isolation on multiplier; revisit architecture |
| Backward | ArchitectureIR | Iterative divider saves area but increases active cycles | INFO | Document trade-off in `design_spec.md` |

The backward feedback is emitted as `ConstraintFeedback` objects and collected in a design-issue report. The agent (or human) must then choose one of the suggested resolutions and update the higher-layer constraints.

## 6. Bidirectional Constraint Flow & Design Gates

### 6.1 Forward Flow: Refinement

```
SpecIR ──► BehaviorIR ──► CycleIR ──► ArchitectureIR ──► StructuralIR ──► DSL ──► Verilog
   │           │            │              │                 │           │          │
   │           │            │              │                 │           │          │
   ▼           ▼            ▼              ▼                 ▼           ▼          ▼
 human       human/ai      human/ai       human/ai          human/ai    ai         ai
constraints  reference    protocol      microarchitecture  structure   RTL       testbench
```

Forward propagation transforms abstract requirements into progressively more concrete constraints and implementations.

### 6.2 Backward Flow: Validation & Feedback

```
Verilog ──► DSL ──► StructuralIR ──► ArchitectureIR ──► CycleIR ──► BehaviorIR ──► SpecIR
   │          │           │                │              │            │            │
   │          │           │                │              │            │            │
   ▼          ▼           ▼                ▼              ▼            ▼            ▼
 synthesis   lint       area estimate    power model    timing       sim          requirement
  results   issues      / floorplan      / clock gate   analysis    coverage      relaxation
```

Backward propagation validates whether the implementation can satisfy the constraints at each upper layer. When it cannot, structured feedback is generated.

### 6.3 Design Gates

Between each pair of layers, introduce an explicit **Design Gate** that checks feasibility before proceeding:

```python
class DesignGate:
    def __init__(self, src_layer: str, dst_layer: str, checks: List[Callable]):
        self.src_layer = src_layer
        self.dst_layer = dst_layer
        self.checks = checks

    def evaluate(self, entity: IREntity) -> List[ConstraintFeedback]:
        issues = []
        for check in self.checks:
            issues.extend(check(entity, self.src_layer, self.dst_layer))
        return issues
```

Example gates:

| Gate | Checks |
|------|--------|
| SpecIR → BehaviorIR | Are all requirements expressible as executable reference behavior? |
| BehaviorIR → CycleIR | Does reference behavior fit within latency/throughput budget? |
| CycleIR → ArchitectureIR | Is the chosen protocol compatible with the target frequency? |
| ArchitectureIR → StructuralIR | Does microarchitecture fit area/power estimate? |
| StructuralIR → DSL | Are all interfaces resolvable without combinational loops? |
| DSL → Verilog | Does generated Verilog pass lint and synthesize? |
| Verilog → SpecIR | Does synthesis/implementation meet original PPA requirements? |

If a gate emits `BLOCKER` feedback, the flow must stop and require resolution before continuing.

### 6.4 Resolution Workflow

When feedback is received, the agent follows a standardized resolution workflow:

1. **Classify**: Is the feedback INFO, WARNING, VIOLATION, or BLOCKER?
2. **Trace**: Which upper-layer constraint(s) are impacted? (`source_constraint_uid`)
3. **Propose**: Use `suggested_resolutions` to generate options.
4. **Decide**: Human or agent selects a resolution path.
5. **Update**: Modify the appropriate layer's constraint(s) and re-propagate.
6. **Re-validate**: Re-run the gate that failed.

```python
def resolve_feedback(feedback: ConstraintFeedback, entity: IREntity) -> bool:
    if feedback.severity == FeedbackSeverity.BLOCKER:
        # Require explicit resolution before continuing
        return apply_resolution(entity, feedback.suggested_resolutions[0])
    elif feedback.severity == FeedbackSeverity.VIOLATION:
        # Log and allow override with human approval
        log_issue(feedback)
        return False
    else:
        # INFO / WARNING: document only
        log_issue(feedback)
        return True
```

## 7. Agent Workflow Standardization & Scaffolding

This framework is not only a technical library; it is a **scaffold that standardizes how RTLCraft agents perform design**. Every agent must follow the same layered workflow with explicit decision records and feedback handling.

### 7.1 Standard Agent Design Loop

```
┌─────────────────┐
│ 1. Read SpecIR  │  Human/AI define requirements and constraints
└────────┬────────┘
         ▼
┌─────────────────┐
│ 2. Propose      │  Agent proposes architecture decisions
│    Architecture │  at BehaviorIR/CycleIR/ArchitectureIR
└────────┬────────┘
         ▼
┌─────────────────┐
│ 3. Forward      │  Propagate constraints down to Verilog
│    Propagate    │
└────────┬────────┘
         ▼
┌─────────────────┐
│ 4. Generate     │  Emit RTL, UVM/cocotb, SVA, reports
│    Artifacts    │
└────────┬────────┘
         ▼
┌─────────────────┐
│ 5. Validate     │  Run tests, lint, synthesis estimates
│    Backward     │
└────────┬────────┘
         ▼
┌─────────────────┐     ┌─────────────────┐
│ 6. Feedback?    │────►│ Yes: resolve and│
│    Issues?      │     │ re-propose      │
└─────────────────┘     └─────────────────┘
         │ No
         ▼
┌─────────────────┐
│ 7. Commit       │  Generate decision records and review bundle
└─────────────────┘
```

### 7.2 Decision Records

Every human- or AI-made design decision that affects constraints must be recorded:

```python
@dataclass
class DesignDecision:
    uid: str
    layer: str
    topic: str
    decision: str
    rationale: str
    alternatives_considered: List[str]
    impacted_constraints: List[str]
    owner: str  # "human" or "ai"
```

Decisions are emitted to `earphone/specs/XX_decision_log.md` and referenced by constraints via `metadata["decision_uid"]`.

### 7.3 Scaffolding Templates

To enforce consistency, provide agent scaffolding templates:

```python
# rtlgen/scaffold.py  (new)
class DesignScaffold:
    def __init__(self, propagator: ConstraintPropagator):
        self.propagator = propagator
        self.gates: List[DesignGate] = []
        self.decisions: List[DesignDecision] = []
        self.feedback: List[ConstraintFeedback] = []

    def add_constraint(self, entity: IREntity, constraint: IRConstraint):
        entity.add_constraint(constraint)

    def refine(self, entity: IREntity, path: List[Tuple[str, str]]) -> List[IRConstraint]:
        return self.propagator.propagate_all(entity, path)

    def validate(self, entity: IREntity) -> List[ConstraintFeedback]:
        for gate in self.gates:
            self.feedback.extend(gate.evaluate(entity))
        return self.feedback

    def record_decision(self, decision: DesignDecision):
        self.decisions.append(decision)

    def generate_review_bundle(self, out_dir: str):
        # Emit constraints, decisions, feedback as markdown
        ...
```

### 7.4 Agent Compliance Checklist

An agent is considered compliant with the scaffold if it:

- [ ] Attaches all requirements as `IRConstraint` at SpecIR.
- [ ] Records all architecture decisions as `DesignDecision`.
- [ ] Propagates constraints forward through all 6 layers.
- [ ] Runs all `DesignGate` checks and handles feedback.
- [ ] Does not silently waive `BLOCKER` or `VIOLATION` feedback.
- [ ] Generates traceability report linking artifacts to constraints.
- [ ] Distinguishes `owner="human"` from `owner="ai"` constraints.

### 6.3 `generate_verification_artifacts()`

```python
def generate_verification_artifacts(entities: List[IREntity], out_dir: str):
    artifacts = {
        "sva": [],
        "uvm_sequences": [],
        "uvm_scoreboards": [],
        "cocotb": [],
        "reports": [],
    }

    for e in entities:
        for c in e.constraints_by(layer="Verilog"):
            if c.category == "functional":
                artifacts["sva"].append(emit_sva(c))
            elif c.category == "verification":
                kind = c.metadata.get("kind")
                if kind == "sequence":
                    artifacts["uvm_sequences"].append(emit_uvm_sequence(c))
                elif kind == "scoreboard":
                    artifacts["uvm_scoreboards"].append(emit_uvm_scoreboard(c))
            elif c.category in ("performance", "power", "timing"):
                artifacts["reports"].append(emit_perf_report(c))

    return artifacts
```

## 9. Human vs AI Value in the Framework

| Layer | Human Value | AI Value |
|-------|-------------|----------|
| **SpecIR** | Define requirements, PPA budgets, safety/compliance boundaries, trade-off priorities | Format requirements, suggest constraints, detect ambiguity |
| **BehaviorIR** | Choose golden reference model, define equivalence semantics, handle undefined behavior | Implement ISS/functional models, generate L1 tests |
| **CycleIR** | Design protocols, latency/throughput trade-offs, clock/reset domains | Generate protocol FSMs, timing checkers |
| **ArchitectureIR** | Innovate microarchitecture, choose PPA optimization strategy | Apply design patterns, generate RTL skeletons |
| **StructuralIR** | Decide module boundaries, interface contracts, reuse strategy | Instantiate modules, wire submodules, insert ICGs |
| **Verilog** | Review code style, optimize critical paths, waive lint | Generate Verilog, run lint, generate testbenches |
| **Meta** | Validate that "tests passing" means "design correct" | Execute tests, report coverage, flag conflicts |

The framework makes this boundary explicit via the `owner` field:

- `owner="human"`: constraints/requirements/decisions that must be reviewed and approved by engineers.
- `owner="ai"`: derived constraints and generated implementations that can be regenerated automatically.

## 11. Success Criteria

1. **Functional**: All existing tests still pass (L1 4/4, L3 4/4, cross-layer 1/1, Verilog 9/9).
2. **Forward constraint coverage**: At least 3 constraints traced through all 6 layers.
3. **Backward feedback coverage**: At least 1 artificial constraint violation is detected, reported, and resolved through the feedback loop.
4. **Artifact generation**: At least 1 UVM sequence, 1 SVA assertion, and 1 cocotb test generated from constraints.
5. **Traceability**: `09_constraint_traceability.md` and `10_design_issues.md` generated and human-readable.
6. **Genericity**: Framework code has no Earphone-specific hard-coding.
7. **Agent compliance**: Earphone flow follows the standardized `DesignScaffold` loop and records all major architecture decisions.

## 12. Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Modifying `rtlgen/core.py` breaks existing designs | High | Start with monkey-patch in `earphone/`; upstream only after stabilization |
| Constraint explosion makes propagation slow | Medium | Cache propagated constraints; use lazy evaluation |
| Human-defined constraints are incomplete | High | Add coverage-gap analysis in traceability report |
| AI-derived constraints conflict | Medium | Add conflict-detection and human-review gate |
| Generated UVM/SVA quality is poor | Medium | Keep human review step; use as skeleton, not final code |
| Agents ignore BLOCKER feedback and continue | High | Enforce `DesignGate` hard stops in `DesignScaffold`; require explicit override flag |
| Feedback loop creates infinite oscillation | Medium | Limit iterations; require human approval for spec relaxation |
| Scaffolding too rigid for novel architectures | Medium | Allow agents to propose new gates/decision types, recorded as meta-decisions |

## 13. Open Questions

1. Should constraints be mutable after creation, or frozen like dataclasses?
2. Should the framework support formal methods (e.g., SymbiYosys properties) as a separate `FormalIR` layer?
3. How should power/area constraints be validated without running synthesis? Should we integrate Yosys area estimates?
4. Should human review be a gating step before AI-derived constraints are propagated to Verilog?
5. How do we prevent agents from gaming the feedback loop by relaxing constraints instead of improving design?
6. Should `DesignGate` checks be mandatory or advisory by default?
