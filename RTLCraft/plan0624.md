## 1. 为什么需要 `plan0624`

`plan0623.md` 已经把 `rtlgen_x` 的主方向收得比较清楚：

1. 以 DSL 作为唯一公开 authoring surface
2. 强化多时钟 / reset / CDC
3. 提升 emitted RTL 的可读性
4. 明确 `archsim` 是轻量架构探索器，而不是高保真 SoC 自动建模器

但随着这轮推进，我们已经得到三个新的、更具体的判断：

1. **行为级仿真的价值是架构探索与软硬件协同，不是替代详细 RTL 仿真**
2. **如果要让生态真正有价值，需要一个真实旗舰设计来牵引标准库和工具闭环**
3. **如果要做 runtime / 驱动 / 编译器 / 算子优化，需要额外的软件栈，但不应该再造一个厚重的新“总控框架”**

所以 `plan0624.md` 的任务不是推翻 `plan0623.md`，而是：

1. 继承 `plan0623.md` 里仍然成立的技术主线
2. 明确哪些内容尚未完成，需要继续执行
3. 把旗舰设计和软件栈正式纳入计划，而不是停留在讨论层

## 2. 总体结论

这一轮之后，我们的总体路线应当明确为：

1. **`rtlgen_x` 继续只做硬能力栈**
2. **`gpu_sm` 从“ worked example ”逐步演化为 GPGPU 旗舰主线的种子**
3. **新增一个独立的软件栈层，但不新增一个大一统 orchestration framework**

一句话说：

**硬件能力栈、旗舰设计、软件栈三条线并行推进，但边界清楚、接口清楚、职责清楚。**

## 3. 对关键问题的正式判断

### 3.1 行为级仿真的价值在哪里

在当前框架里，行为级仿真的价值主要落在 `rtlgen_x.archsim`，而不在 DSL 详细执行器本身。

它的核心价值不是证明 RTL 细节正确，而是更早回答：

1. 当前方案的主要瓶颈是什么
2. 瓶颈是带宽、容量、II、队列还是时延
3. 哪个架构 knob 最值得先改
4. 编译器 / runtime / workload 映射是否和硬件结构匹配

因此它在框架中的角色应当明确为：

1. **行为级仿真**
   - 快速 throughput reasoning
   - 快速 sweep
   - 粗粒度 workload-to-architecture 匹配
2. **周期级仿真**
   - backpressure / queue pressure / contention
   - stage occupancy / utilization
   - 更可信的 bottleneck ranking
3. **DSL / RTL 仿真**
   - 详细实现正确性
   - reset / CDC / protocol / reference-model 闭环

所以正确链路不是：

```text
behavior sim -> 自动生成 DSL
```

而是：

```text
workload / kernel shape
  -> behavior-level archsim
  -> cycle-level archsim
  -> report / sweep / bottleneck ranking
  -> agent or designer edits DSL / RTL
  -> detailed sim / cosim / UVM / PPA
```

### 3.2 标准库和旗舰设计的关系

如果没有旗舰设计，标准库很容易退化成“看起来通用，但闭环不足”的对象堆积。

因此标准库建设必须改成：

1. **由真实设计牵引**
2. **从真实设计中抽象**
3. **只有完成闭环的对象才进入 stable stdlib**

GPGPU 作为旗舰设计是合理的，因为它天然覆盖：

1. 吞吐型架构探索
2. 多时钟 / buffer / arbitration / shared memory
3. valid/ready / DMA / controller / scoreboard / command queue
4. 编译器 / runtime / workload 对硬件结构的反向牵引
5. PPA / archsim / CDC / verify 的真实用例

### 3.3 runtime / 驱动 / 编译器 / 算子优化是否需要额外框架

结论是：

1. **需要额外的软件栈**
2. **不需要额外的厚重框架**

也就是说，我们应该增加的是：

1. runtime
2. driver
3. kernel launch ABI
4. compiler / lowering
5. operator library / autotuning
6. workload trace / perf analysis

但这些不应当塞回 `rtlgen_x` 的本体，也不应当再包一层大 orchestration。

更好的方式是：

1. `rtlgen_x` 保持为硬件能力栈
2. 软件栈独立成并列目录或并列 repo
3. 二者通过少量明确工件连接

连接工件应当包括：

1. CSR / MMIO map
2. command descriptor format
3. kernel metadata
4. memory layout / address map
5. perf counter schema
6. workload trace format

## 7. 主轴 B：GPGPU 旗舰设计计划

## 7.1 为什么是 GPGPU

GPGPU 是这套框架当前最能体现价值的旗舰方向，因为它同时需要：

1. 架构探索
2. 详细 DSL / RTL
3. 多时钟 / memory / CDC
4. reference model / cosim / UVM
5. PPA 分析
6. workload / compiler / runtime 协同

## 7.2 旗舰范围定义

这里不做 graphics GPU，而是做 **compute-oriented GPGPU accelerator chip**。

第一阶段范围建议为：

1. command processor
2. 1 个可工作的 SM / warp cluster
3. SIMD ALU
4. SFU
5. load/store + shared memory / scratchpad
6. DMA / memory interface
7. perf counters
8. 最小 kernel launch 模型

第二阶段再扩展到：

1. 多 SM cluster
2. 更完整的 interconnect / fabric
3. cache / shared-memory 更复杂配置
4. 更强的 warp scheduling / scoreboard / hazard logic
5. 更复杂的 operator / tensor-like datapath

## 7.3 为什么不直接做完整大 GPU

因为当前框架最需要的是：

1. 一个真实闭环项目
2. 能持续反向驱动 stdlib / verify / archsim 的项目
3. 范围可控、问题密度高的项目

所以更合适的是：

1. **先把 `gpu_sm` 从 worked example 演进成旗舰种子**
2. **再从 `SM` -> `cluster` -> `chip` 逐步扩张**

## 7.4 GPGPU 旗舰应当反向牵引什么

### 对 stdlib 的牵引

1. valid/ready bundle
2. queue / scoreboard / arbiter / issue helpers
3. shared-memory / banked storage helper
4. DMA / command queue / CSR helper
5. verification VIP

### 对 archsim 的牵引

1. warp workload 建模
2. occupancy / issue / writeback pressure
3. shared-memory / coalescer / SFU / writeback 争用
4. behavior vs cycle 的价值比较

### 对 PPA 的牵引

1. lane count tradeoff
2. shared-memory banking / queue depth / writeback width
3. multiplier / SFU hotspot
4. memory-heavy and lane-heavy area/power proxy

## 8. 主轴 C：软件栈计划

## 8.1 这不是另一个“框架”

软件栈需要，但不应该变成另一个厚框架。

正式边界应当是：

1. **`rtlgen_x`**
   - DSL
   - emit
   - sim
   - verify/UVM
   - CDC
   - PPA
   - archsim
2. **软件栈**
   - runtime
   - driver
   - compiler
   - operator library
   - autotuning / workload tooling

二者之间只通过接口工件连接。

## 8.2 软件栈最小范围

第一阶段的软件栈不求完整，只求能支撑旗舰设计闭环：

1. kernel metadata schema
2. command descriptor schema
3. launch/runtime ABI
4. memory layout contract
5. workload trace format
6. 最小 driver/runtime stub

## 8.3 软件栈建议结构

建议结构是一个独立目录，例如：

```text
gpgpu_stack/
  README.md
  abi/
  runtime/
  driver/
  compiler/
  operators/
  traces/
```

这里的重点不是把内容一次做满，而是先把边界和接口固定。

## 8.4 软件栈如何与硬件主线配合

推荐的闭环是：

```text
kernel/operator intent
  -> compiler/lowering
  -> workload trace + command descriptors
  -> behavior-level archsim
  -> cycle-level archsim
  -> DSL/RTL refinement
  -> verify/cosim/UVM
  -> perf counters / reports
  -> runtime/compiler retuning
```
