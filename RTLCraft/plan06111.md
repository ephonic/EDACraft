# RTLCraft 性能驱动 Verify 分层落地计划

日期：2026-06-11

## 1. 背景

当前项目已经开始具备多层合同化建模的雏形：

- `ModuleContract`
- `FunctionalObjective`
- `PerformanceObjective`
- `VerificationHook`
- `contract_emit` stage

但现在 `verify` 仍然主要停留在：

- requirement 是否存在
- skeleton / DSL 是否完整
- 是否存在 undriven output

这意味着系统仍然可以验证“结构是否像样”，却不能验证“局部性能是否达标”。

如果我们希望这个框架从 toy 提升到专业设计生成框架，那么性能不能只存在于：

- 描述性文档
- PPA 优化阶段的参考目标
- 最终人工阅读

而必须进入 `verify`，成为每一层、每个子模块都可以局部判定的显式约束。


## 2. 核心目标

把 `verify` 从单纯的结构/功能检查扩展为：

```text
contract_check
  + functional_check
  + perf_static_check
  + perf_dynamic_check
  + diagnosis
```

也就是每个模块在 verify 阶段都必须回答：

1. 合同是否存在
2. 功能目标是否满足
3. 局部性能目标是否满足
4. 若不满足，最可能的局部瓶颈在哪里
5. 哪些 `optimization_knobs` 被允许用于下一轮局部优化


## 3. 为什么性能要在 verify 而不是只在 optimize

如果性能只存在于优化阶段，那么优化器只能“猜哪里值得改”。

如果性能进入 verify：

- verify 会明确指出哪个模块失败
- 每个失败都对应一个局部性能目标
- diagnosis 会把失败映射到允许修改的 knob
- 后续 lowering / repair / agent refinement 就能在局部闭环下进行

因此：

- optimize 负责尝试改进
- verify 负责判定是否达标

没有 verify 中的性能判据，优化就缺少收敛目标。


## 4. 设计原则

### 4.1 先局部，再全局

先不要直接在 verify 中追：

- 全核 IPC
- 全系统 CPI
- 大程序 benchmark 分数

第一阶段只做模块局部性能验证，例如：

- `FrontendUnit`：redirect latency、fetch 有效率
- `DecodeRenameUnit`：rename 延迟、rename 吞吐
- `BackendUnit`：dispatch 接收率、ROB backpressure
- `CommitUnit`：retire 可见性延迟、连续 retire 能力
- `ClusterTop`：core 到 private L1 的请求通路
- `MeshTop`：grant latency、probe fanout latency

这样失败会被局部化，生成和优化也更容易被约束住。

### 4.2 性能目标必须是机器可检查的

不能只保留：

- `"1 cycle latency"`
- `"high throughput"`

这种文本描述。

必须增加结构化性能检查项，供 verify 自动执行。

### 4.3 性能检查要分层

不同层的检查方式不同：

- L1/L2：行为模型层的性能规则和顺序约束
- L3：DSL 局部结构 + 轻量动态仿真
- L4：更精确的 pipeline cut / queue / reg 位置检查
- L5：lint / synthesis / timing-style 静态检查

当前优先把 L3 verify 做起来。


## 5. 建议的数据结构扩展

在 `rtlgen/contracts.py` 中，为 `PerformanceObjective` 增加可执行检查项。

建议增加：

- `PerfCheck`
- `PerfScenario`
- `PerfCheckResult`
- `ModulePerfReport`

### 5.1 PerfCheck

每一个局部性能检查的最小单元。

建议字段：

- `name`
- `kind`
- `description`
- `source_signals`
- `sink_signals`
- `max_cycles`
- `min_rate`
- `max_ratio`
- `budget_key`
- `sample_cycles`
- `required_value`

其中 `kind` 初期建议支持：

- `latency`
- `throughput`
- `occupancy`
- `stall_ratio`
- `duty_cycle`
- `event_count`
- `structural_budget`

### 5.2 PerfScenario

定义一个短小、局部、可重复的输入激励场景。

建议字段：

- `name`
- `description`
- `reset_cycles`
- `cycles`
- `stimulus`
- `expected_observations`

第一阶段不追求通用脚本语言，只支持简单 stimulus 描述：

- 常量驱动
- 固定周期内拉高某输入
- 指定某信号持续为 1

### 5.3 PerfCheckResult

记录单个检查的执行结果：

- `name`
- `kind`
- `passed`
- `observed`
- `expected`
- `details`

### 5.4 ModulePerfReport

记录模块级性能验证结果：

- `module_name`
- `contract_present`
- `static_checks`
- `dynamic_checks`
- `passed`
- `blocking_failures`
- `recommended_knobs`


## 6. Verify 中的性能检查分层

### 6.1 perf_static_check

基于结构和 PPA 分析快速检查：

- `max_logic_depth`
- `rob_depth`
- `queue count`
- `counter width`
- `issue queue budget`
- `allowed_submodules` 合规性

主要输入：

- `ModuleContract.performance_objective.structural_budget`
- `PPAAnalyzer` 报告
- module AST / submodule 数量

目的：

- 快速发现“结构上已经不可能满足目标”的情况

### 6.2 perf_dynamic_check

基于 `Simulator` 做轻量局部仿真。

检查类型示例：

- latency check:
  - 触发信号出现后，响应信号必须在 `N` 个 cycle 内出现
- throughput check:
  - 在 `K` 个 cycle 持续激励下，输出事件总数必须 >= 某个门槛
- stall ratio:
  - `stall/full/backpressure` 信号占比不得超过上限
- duty cycle:
  - `valid/req` 信号有效占空比不得低于下限

### 6.3 diagnosis

verify 不只输出 pass/fail。

应同时输出：

- fail 的 check 名称
- 观测到的值
- 预期值
- 可能关联的局部瓶颈
- 可调整的 `optimization_knobs`

例如：

```text
BackendUnit:
  fail: dispatch_accept_rate
  observed: 0.42
  expected: >= 0.80
  suspect: ROB backpressure / issue_queue_partitioning
  knobs: rob_depth, issue_queue_partitioning
```


## 7. 第一阶段试点模块

先只在这几个模块试点：

1. `FrontendUnit`
2. `DecodeRenameUnit`
3. `BackendUnit`
4. `CommitUnit`
5. `OoOCore`
6. `ClusterTop`
7. `MeshTop`

原因：

- 这些模块已经有 `ModuleContract`
- 它们覆盖 core 内部和系统集成两个层次
- 适合验证“局部性能驱动 + 局部优化约束”的方法


## 8. 推荐的首批性能检查

### 8.1 FrontendUnit

- `fetch_enable_latency`
  - reset 释放后，`fetch_valid` 在 1 cycle 内可见
- `fetch_request_duty_cycle`
  - 初始化后 `icache_req` 占空比应高

### 8.2 DecodeRenameUnit

- `rename_response_latency`
  - `fetch_valid` 拉高后 `rename_done` 在 1 cycle 内出现
- `rename_throughput`
  - 持续激励下 `rename_done` 平均每拍至少 1 次

### 8.3 BackendUnit

- `dispatch_to_retire_activity`
  - 持续 `dispatch_valid` 时应出现 retire activity
- `rob_backpressure_ratio`
  - `rob_full` 占比不得长期过高

### 8.4 CommitUnit

- `retire_visibility_latency`
  - `retire_in` 到 `retire_valid` 的延迟
- `stall_mirroring`
  - `rob_full` 到 `core_stall` 的可见性

### 8.5 OoOCore / ClusterTop / MeshTop

第一阶段主要做静态结构检查：

- 子模块数量
- memory hierarchy attachment
- top-level allowed_submodules 合规性

动态检查先保持轻量，避免被系统级复杂性拖慢。


## 9. 代码修改顺序

### Step A. 扩展合同 IR

文件：

- `rtlgen/contracts.py`

工作：

1. 增加 `PerfCheck`
2. 扩展 `PerformanceObjective`
3. 增加 `PerfCheckResult`
4. 增加 `ModulePerfReport`

### Step B. 给试点模块填入可执行 perf checks

文件：

- `rtlgen/cpu_gen.py`
- 必要时 `skills/riscv_ooo_4core/dsl_modules.py`
- 必要时 `skills/riscv64_soc/dsl_modules.py`

工作：

1. 为 `FrontendUnit / DecodeRenameUnit / BackendUnit / CommitUnit` 增加 `checks`
2. 先以：
   - latency
   - throughput
   - structural_budget
   为主

### Step C. 在 verify 中加入 perf 验证器

文件：

- `rtlgen/skill_ppa.py`

工作：

1. 新增 `_run_perf_verify_module()` 之类 helper
2. 在 `_run_verify()` 中加入：
   - contract presence
   - perf static checks
   - perf dynamic checks
   - perf diagnosis
3. 输出：
   - `verify/<module>.perf.json`
   - 汇总 metrics

### Step D. 温和 gate

第一阶段只做：

- 缺合同：verify fail
- 有合同但 perf check 未定义：warning
- perf check 执行后 fail：verify fail，但不阻塞 `rtl` stage

第二阶段再收紧为：

- 无 perf checks 的模块不得进入深度 lowering / agent refine


## 10. 验收标准

第一阶段完成时，应满足：

1. `contract_emit` 输出的 JSON 中包含结构化 perf checks
2. `verify` 会对试点模块生成 perf report
3. report 中包含：
   - pass/fail
   - observed
   - expected
   - recommended knobs
4. `riscv_ooo_4core` 和 `riscv64_soc` 仍能通过 `rtl -> lint`
5. 至少一个模块的 verify 已经不再只是结构检查，而是真正包含性能判据


## 11. 长期方向

下一阶段可以继续扩展：

- 从 L3 局部性能验证扩展到 L2/L3 cross-check
- 把行为模型中的事件定义映射到 perf checks
- 把 `agent_rtl_gen` 和 `repair` 直接绑定到 perf diagnosis
- 引入更真实的 NoC / cache / ROB 压力场景
- 最终形成：

```text
合同定义目标
  -> verify 判定局部达标
  -> diagnosis 定位局部问题
  -> optimize / repair 只改允许的 knobs
  -> 再次 verify 收敛
```

这才是“性能和功能共同驱动、约束在局部”的专业生成框架。
