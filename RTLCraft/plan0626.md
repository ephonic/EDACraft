## 5. 第一优先级：P0 真实 blocker 收口

### P0-A. JPEG IDCT signed rounding / clip / width composition 问题

#### 2026-06-26 进展

已完成首轮代码收口并通过 targeted regression：

1. Python runtime 已补齐 signed `+/-/*` 的按位宽解释，不再把 two's-complement state 当无符号 payload 直接参与算术
2. compiled C++ backend 的 scalar / wide 路径都已补齐 signed arithmetic codegen，修复了 nested `RoundShiftRight(...)` 链路里 `acc + prod` 被 C++ unsigned promotion 污染的问题
3. `audit0626.md` 点名的 JPEG gradient `(0,7)` 饱和到 `255` 的问题已消失
4. 现有回归已覆盖：
   - `jpeg_decoder/tests/test_cpp_idct.py`
   - `rtlgen_x/tests/test_dsl_import.py -k jpeg_idct_signed_level_shift_matches_reference_and_compiled`
   - `rtlgen_x/tests/test_cpp_backend.py` 中新增的 nested signed round-shift regression
   - `rtlgen_x/tests/test_python_runtime.py` 中新增的 signed-add-before-round-shift regression

#### 收口结果

1. `audit0626.md` 里点名的 `(0,7)` gradient failure 已消失
2. JPEG IDCT / decoder 相关 reference regression 已覆盖 Python simulator 与 compiled simulator
3. signed arithmetic / round-shift / level-shift / clip 已有独立 DSL regression，避免只靠 JPEG 大测试兜底
4. 后续 JPEG fixed-point 新痛点按“先补 reference-driven regression，再落 helper/backend 修复”的方式维护

### P0-B. emitted RTL / backend closure contract 定稿

4. 文档与脚本口径已开始收拢：
   - JPEG cookbook 明确 Python/compiled sim 是主功能闭环路径
   - `jpeg_decoder/tests/test_verilog.py` 明确并强制执行 `iverilog` compile smoke 语义
   - `rtlgen_x/README.md` 已明确 `iverilog` 是本地 smoke backend，而更强 emitted-RTL closure 优先 `verilator` / `vcs`

这一项不再按“默认 iverilog 必须吃下所有 emitted RTL”来理解，而是显式区分：

1. **正式推荐闭环 backend**
   - `verilator`

2. **兼容性 backend**
   - `iverilog`

`audit0626.md` 报出的 reserved identifier / SystemVerilog 兼容性问题已用 emitter regression 与 JPEG compile smoke 锁住。

#### 已完成动作

   - `iverilog` 保留为兼容性 smoke，而不是唯一 correctness gate
2. 保留 `iverilog` regression，并明确它验证的是兼容性 compile smoke subset
3. 对 README / cookbook / support matrix 统一 backend 叙述，避免计划、代码、测试三者口径不一致

#### 验收

1. JPEG emitted RTL 的主闭环测试不再卡在“工具链 contract 不清楚”
2. backend policy 在文档与测试里一致
3. `iverilog` 若继续保留，边界清晰；若不再作为主 gate，也明确写入迁移说明
