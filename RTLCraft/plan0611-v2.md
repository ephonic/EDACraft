# RTLCraft Transaction / Protocol Recipe v2 Plan

日期：2026-06-11

## 1. 当前状态

上一轮已经把局部性能验证从“手写 perf check”推进到了结构化事务约束：

- `TransactionContract` 支持 `recipe`、trigger/completion event、qualifier、`sample_cycles`、metadata
- `PerfScenario` 支持 `linked_transactions` 和 stimulus recipe
- `verify` 能从 transaction recipe 自动派生 perf check
- `BehaviorIR` 和 `VerificationPlanIR` 已经能携带同一份 transaction/protocol boundary
- 真实模块 `CoherenceDir`、`DRAMCtrl` 已经开始使用 recipe 化 transaction

这说明框架已经从“信号级检查集合”进入了“事务合同 + 场景合同 + 事件语义 + 自动 verify”的方向。

## 2. 还需要改进的地方

### 2.1 抽出统一 recipe registry

当前 `ready_valid_transfer`、`request_grant_completion`、`single_outstanding_response` 的语义仍然散在 validate、derive、markdown/IR 投影逻辑里。

应抽成统一 registry，每个 recipe 明确定义：

- recipe 名称
- 必需 signal 形态
- 默认 trigger/completion event
- qualifier 要求
- 自动派生的 perf check
- 可生成的最小 scenario
- 可投影到 BehaviorIR / CycleIR / VerificationPlanIR 的语义摘要

这是最高优先级，因为它会把“代码里的 if 分支”提升成可复用专业模板库。

### 2.2 把 transaction 投影进 CycleIR

当前 transaction 已经进入 BehaviorIR 和 VerificationPlanIR，但 CycleIR 仍主要是文本化 handshake rule。

下一步建议加入：

- `TransactionEventIR`
- `CycleTransactionIR`

目标是让对齐关系变成：

```text
BehaviorIR transaction
  == CycleIR event boundary
  == VerificationPlanIR trace/assert/check
```

### 2.3 让 VerificationPlanIR 反向驱动 verify

目前真正执行 perf verify 的入口仍然主要是 `ModuleContract`。

后续应支持：

- `VerificationPlanIR.transactions` 派生 perf check
- `VerificationPlanIR.assertions` 进入轻量 checker
- `VerificationPlanIR.tracepoints` 驱动仿真观测

这样 sidecar 不只是报告，而是可以成为 verify 输入。

### 2.4 自动合成 transaction scenario

现在 scenario 能 link transaction，但多数 stimulus 仍然需要手写。

recipe registry 后可以自动生成最小 scenario，例如：

- ready/valid accept pulse
- request -> grant -> completion
- single outstanding response
- backpressure window
- outstanding violation probe

### 2.5 固化 schema version / migration

`SpecIR`、`BehaviorIR`、`VerificationPlanIR`、`ModuleContract` 演进很快，需要：

- schema version
- backward-compatible `from_dict`
- migration tests
- sidecar compatibility tests

否则后续多轮 lowering 很容易被旧 JSON 绊住。

### 2.6 扩展真实模块 recipe 覆盖

建议逐步把 cache/noc/arbiter 中的常见约束 recipe 化：

- fairness arbitration
- backpressure hold
- ordered completion
- multi-response transaction
- bounded queue occupancy
- no-drop/no-duplicate stream

目标是进一步减少 agent 自由编写信号级检查的空间。

## 3. v2 推进顺序

1. 建立 `transaction_recipes` registry，收拢已支持的三类 recipe。
2. 让 `skill_ppa.verify` 只调用 registry 派生 perf check。
3. 给 registry 加单元测试，保证 recipe 名称、默认事件、派生 check 不漂移。
4. 下一步扩展 CycleIR transaction event。
5. 再把 VerificationPlanIR 作为可执行 verify 输入。
