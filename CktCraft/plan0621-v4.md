## 1. 问题 #1：S2 grid BSIM4 HB-NL 仍 0/N

### 1.1 现象

- 用例：`tests/test_convergence_grid.cpp::S2_Bsim4CsNhGrid`、`tests/test_large_circuit.cpp::S2 grid`。
- 结果：DC OP 收敛，HB Newton **直接发散 / 0 Newton step accept**。
- 日志关键句（已开启 `autoHomotopy=true`）：
  ```
  HB-NL: sourceSweep ε=1.0 ‖F‖0 = 2.7e-2 → 8.4e-3 → … (前 5 步 OK)
  HB-NL: λ-search exhausted (α=1.0…1e-6), accept fail
  HB-NL: total 0 / 25 Newton accepts → DIVERGED
  ```
- 同套测试，把 BSIM4 替换成 `nmos_sh` (EKV) 或在 NH=1 时跑：均 PASS。证明并非框架级缺陷，而是 **频域 Newton 在 BSIM4 强非线性 + 高 NH 时的算法饱和**。

### 1.2 根因（按影响排序，全部 code-grounded）

#### 1.2.1【高】残差 F 与 Jacobian J 不一致 — 缺失 jωQ 项

- **位置**：`src/assembly/hb_jacobian.cpp:399-402` (residual) vs `:424-439` (Jacobian)。
- **症状**：
  - 残差 `F` 在 line 399-402 只累加电阻性谐波 `-iHarm[k]`：
    ```cpp
    F[base + 2k]     -= iHarm[k].real();
    F[base + 2k + 1] -= iHarm[k].imag();
    ```
  - Jacobian `J` 在 line 424-439 **同时** 累加 `G_pq[k+m] + jω_k·C_pq[k+m]` 的电纳块（jωQ/jωC 项已加入）。
- **后果**：Newton 步永远朝着 J 的"完整切平面"前进，但 F 却以"只有电阻部分"评估，方向 mismatch；最小化 ‖F‖² 时直接撞回起点。
- **历史记录**：`status0620_v2.md:93` 早已留 TODO：
  > "残差也需补 jωQ 项才能闭环；当前只有 J 加了，F 没加。"

#### 1.2.2【高】时域采样数 N=2(NH+1) 偏少 → 卷积混叠

- **位置**：`src/assembly/hb_jacobian.cpp:246`：
  ```cpp
  size_t N = 2 * (NH + 1);   // 仅满足最低 Nyquist
  ```
- **症状**：BSIM4 在饱和/三极管边界附近的等效非线性阶数 K≈5；卷积要求 `N ≥ 2K·NH+1`。当 NH=3，K=5 时 N 应 ≥ 31，实际只有 8；`G[k+m]` 在 `k+m>NH` 处被 **直接截 0**（line 187 内 `if (k+m>NH) continue;`），导致 J 高频块被人为零化，‖F‖ 与 ‖J·Δx‖ 之间出现假平台。
- **附带后果**：FFT 频率分辨率不足，导致 evalTimeSamples 在 1/N 周期处刚好落在 BSIM4 的"内部 V 限位"边界上，触发 limiting 重置（见 1.2.5）。

#### 1.2.3【中】HB warm-start k≥1 谐波全零

- **位置**：`src/solver/hb_nonlinear.cpp:577-588`，warm-start 只填 `X[i][0]` (DC bias)：
  ```cpp
  for (size_t i = 0; i < nNode; ++i) {
      X[i][0] = dcSolution[i];     // 只填 DC
      // X[i][k>=1] 默认 0
  }
  ```
- **症状**：sourceScale=1 时（即真值），驱动 V_in cosωt 直接灌入"全部 k≥1 = 0"的初值，第一次 Newton 步落差等价于"线性 AC 一次解+非线性整流分量"全部由 Newton 承担 → 步长选择困难。
- **横向对比**：DC OP 有 source-step 同伦 + 几何 schedule（dc_op.cpp:240 起）；HB 已有 sourceSweep 框架但仍未利用 AC small-signal 解作 k=1 初值。

#### 1.2.4【中】Armijo line-search 比较式 BUG

- **位置**：`src/solver/hb_nonlinear.cpp:457-462`：
  ```cpp
  if (fTrial <= fNorm * (1.0 - 1e-4 * alpha)) {  // ← 比 |F| 而非 |F|²
      accept;
  }
  ```
- **正确形式**：Armijo 应以 `‖F‖²` 比较，即 `fTrialSq ≤ fNormSq·(1 − 2·1e-4·α)` 或等价以 `‖F‖` 比较时系数减半。当前实现的"减少率门槛"是真值的 2 倍严格，导致小 α 时几乎永远拒绝。
- **后果**：在 1.2.1+1.2.2 已使 F 方向不准的情况下，line-search 再以过严门槛回绝，直接 ε-search exhausted。

#### 1.2.5【中】evalTimeSamples bad-sample gate `|vv|>100` 过松

- **位置**：`src/model/osdi_model.cpp:143-154`：
  ```cpp
  if (std::abs(vv) > 100.0) {
      vv = 0.0;            // 静默零填
      bad = true;
  }
  ```
- **症状**：100 V 的门槛对 1.5 V 系统毫无意义，但 BSIM4 limiting 内部会产生 ±50 V 中间态（特别是 Vbs 自洽未收敛时）。bad=true 时整个 sample 被 0 填，FFT 输入失去周期性 → G/C 时序卷积被引入"方波 artifact"。
- **修复门槛**：应 ≤ 5×电源电压（如 VDD=1.5 V → 门槛 7.5 V），并在触发时回退 limiting 而非静默零填。

#### 1.2.6【低】缺 HB verbose 开关，根因定位困难

- **对比**：`src/solver/dc_op.cpp:16-19` 已有 `RFSIM_DCOP_VERBOSE` 环境变量打印每步 ‖F‖、ratio、α。
- **现状**：HB 路径全程只在最终发散时报错，中间步 ‖F‖ 轨迹、condition number、λ-search 路径全部沉默 → 用户/CI 都难以分辨"算法限制"还是"模型 bug"。

### 1.3 修复路线（按性价比 D→B→A→C）

#### 路径 D【强烈推荐，先做】AC 小信号 warm-start

- **思路**：DC OP 完成后，复用 hb_jacobian 在 ω=ω₀ 上装一次 *线性化 Y(jω₀)*，解一次小信号 AC：`Y(jω₀)·X₁ = -j·V_s`。把 X[i][1] 写为该 AC 解，X[i][k≥2]=0。
- **代码点**：
  - `src/solver/hb_nonlinear.cpp:577-588`（warm-start 区段）
  - 复用 `assembleHbJacobian()` 在 ε=0、k=1 的 block。
- **代价**：100~200 行；不引入新数值方法。
- **预期效果**：S2 grid NH=1~2 立即可收敛；NH≥3 仍需 1.2.2 oversampling。

#### 路径 B【推荐，与 D 同期】4× oversampling + jωQ 残差补全

- **B1**：把 `hb_jacobian.cpp:246` 的 `N` 改为 `N = std::max<size_t>(8, 4*(NH+1))`；同步放开 `:187` 的 `if (k+m>NH) continue;` —— 仅在 |G[k+m]| 极小时再丢。
- **B2**：在 `hb_jacobian.cpp:399-402` 残差累加里补 `+ jω_k·qHarm[k]`，与 :424-439 的 Jacobian 一一对齐。
- **代价**：80~150 行；需要扩 FFT scratch 缓冲（thread_local 已在 P4 落地，可直接放大）。
- **预期效果**：消除 F/J 方向 mismatch；NH=3~4 在 BSIM4 上变得可解。

#### 路径 A【中优】trust-region / LM 自适应 λ

- **思路**：替换当前 Armijo + sourceSweep 双层为 Levenberg–Marquardt：每次 Newton 失败时把 J → J + λI, λ←10λ；接受时 λ←λ/3。
- **附带补丁**：1.2.4 的 Armijo 比较式 BUG 顺手修正（改为 ‖F‖² 比较或 ‖F‖ 比较时系数减半）。
- **代价**：80~300 行；需缓存 Jᵀ·F 的 norm 历史。
- **预期效果**：路径 D+B 已能解决 S2 grid 主要 case；A 是"防御性兜底"，让 NH=5、Vds>2 V 这类硬 case 也能收敛。

#### 路径 C【低优，可选】频率 ramp 同伦

- **思路**：从 ω₀/10 开始解 HB，每步 ω×1.3 直到目标频率。每步用上步解 warm-start。
- **代价**：150~250 行；需要在 hb_nonlinear 引入 outer-loop 控制。
- **预期效果**：D+B+A 之后基本不需要；仅留作 RF 极端非线性 (Class-E PA, mixer) 的后手。

#### 正交补丁（同期落地，工作量小）

1. **Armijo 比较式修正**：hb_nonlinear.cpp:457-462 改为 `fTrialSq ≤ fNormSq·(1 − 2e-4·α)`。
2. **bad-sample gate 收紧**：osdi_model.cpp:143-154 把门槛从 100 V 改为 `5 * max(|V_supply|)`；触发时记录 warn 而非静默零填。
3. **`RFSIM_HBNL_VERBOSE` 环境开关**：对齐 `RFSIM_DCOP_VERBOSE`，打印每个 Newton step 的 ‖F‖、ratio、α、accept/reject。
4. **sourceScale 行为复核**：hb_jacobian.cpp:339-340 当前把 sourceScale 同时乘到 k=0 (DC) 和 k=1 (AC mag)；DC 部分应在 sourceSweep 之外固定为 1，否则 ε<1 时 DC bias 也被错误压缩。

### 1.4 验收

新增 `tests/test_hb_bsim4_grid.cpp`（或扩展现有 S2 grid）：
- NH=1, Vds∈{0.5, 1.0, 1.5} V — 必须 PASS。
- NH=2, Vds∈{0.5, 1.0, 1.5} V — 必须 PASS。
- NH=3, Vds∈{0.5, 1.0} V — 必须 PASS。
- NH=3, Vds=1.5 V — 允许标记 KNOWN-LIMITATION 但有理由码（路径 A 未上时）。

## 2. 问题 #2：N≥15 BSIM4 多实例段错误/挂起

### 2.1 现象

- 用例：`tests/test_multi_device.cpp::Bsim4ManyInstance_*`、`test_large_scale.cpp::CascodeChain*`。
- 分组：A1/A2 单跑 PASS；与其他 HEAVY 同跑或 N≥15 时 segfault（`status0620_v4.md:96-110`）。
- C3-bis 二分（test_multi_device.cpp:746-884）显示：**不是单纯实例计数**，而是当 *第 3 个* BSIM4 实例的 NodeId=4 与已有实例冲突时崩溃 → 堆累积签名。

### 2.2 根因

#### 2.2.1【已证实，最主要】跨 CRT 堆腐败

- **直接证据**：`src/model/osdi/osdi_library.cpp:32-37` 已有显式注释：
  > "freeing across the CRT boundary corrupts the heap, so we intentionally leak the message buffer."
- **来源**：
  - `build_model.bat:10-14` 用 `vcvars64.bat` → bsim4.dll 由 **MSVC** 链接 `msvcr*.dll/ucrtbase.dll`。
  - 主程序由 **MinGW64 GCC 15.2** 链接 `msvcrt.dll`（默认）或 ucrt（视配置）。
  - 任何由 dll 内 `malloc` 的指针，传到 host 后 `free`/`delete` → 立即堆双链表破坏。
- **当前缓解**：osdi_library.cpp 已把 *message buffer* 故意泄漏；但是 OSDI 0.3 spec 里还有其它跨边界对象：`osdi_descriptor_t` 字段、`setup_model` 内 strdup 的参数表、`evaluate` 返回的中间向量等，未全部锁死。
- **N≥15 触发临界**：每个 BSIM4 实例 setup_model 都至少 alloc 一次 model param table + node-map shadow；当 alloc 次数累积超过 ~14（heap 段刚好覆盖某个跨 CRT free 残留），下一次 alloc 落到坏链表 entry 上 → segfault。

#### 2.2.2【部分已缓解】OsdiModelBlock setup_model 状态共享

- **位置**：`src/model/osdi/osdi_client.cpp:136-192`，V2-γ C3 修复后 modelBlock 通过 `block->setup` 标志防止 setup_model 重复调用。
- **残留风险**：setup_model 在第一次调用时分配的 model-level 静态表（如 BSIM4 内部 process corner 缓存），如果 OSDI 实现内部用全局变量持有，**所有实例共享**；任一实例修改即"串拥"。
- **NodeId=4 特异性**：C3-bis 显示问题集中在第 3 实例 NodeId=4 配置 → 暗示 BSIM4 内部某个数组用 `NodeId-1` 索引，当 NodeId=4 且实例数 ≥3 时触发越界写。这部分**无法在 host 端修**，只能：
  - 短期：用 ekv/nmos_sh 边界验证（路径 C）。
  - 中期：用 GDB + Application Verifier PageHeap 抓现场（见 2.4）。

#### 2.2.3【已排除】nodeMap 悬空指针

- **位置**：`src/model/osdi/osdi_client.cpp:235-240`：
  ```cpp
  instData_.node_mapping.assign(map.begin(), map.end());  // COPY
  ```
- **结论**：是深拷贝，不存在悬空。早期假设作废。

#### 2.2.4【已排除】per-instance scratch 共享

- **位置**：osdi_client.cpp:311-317 (`jacScratch`)、:293-302 (`dcPrevState_/dcNextState_`)：均是 per-instance 成员，无共享。

### 2.3 修复路线

#### 路径 D【立即可做】文档化 KNOWN-LIMITATION + SKIP gate

- **目标**：让 CI 不再红，但风险公开。
- **动作**：
  1. 在 `README.md` / `docs/known_issues.md` 新增章节："BSIM4 OSDI dll 多实例 N≥15 在 cross-CRT 环境下不稳定"。
  2. `tests/test_multi_device.cpp` 已有 `RFSIM_TEST_HEAVY=1` gate，再加一层 `RFSIM_TEST_BSIM4_MULTI=1`；默认 SKIP，CI 仅在 nightly 跑。
  3. 在 `tests/CMakeLists.txt` 把 N≥15 BSIM4 case 打标 `[heavy][cross-crt]`，方便筛选。
- **代价**：30~50 行 + 1 个 doc。
- **效果**：开发体感 100/100 → 维持；KNOWN-LIMITATION 显式。

#### 路径 C【短期推荐】用 ekv/nmos_sh.dll 替代做边界验证

- **目标**：证明算法层无 bug，bug 仅在 BSIM4.dll 自身。
- **动作**：把 `Bsim4ManyInstance_*` 复制为 `Nmos_shManyInstance_*`，参数等效缩放，N=20、N=50 跑通 → 写入 status 文档作为"OSDI runtime + circuit 框架在多实例下正确"的证据。
- **代价**：50~100 行；纯测试代码。
- **效果**：缩小怀疑面，避免误改 osdi_client / circuit。

#### 路径 A【中长期】MinGW 重编 bsim4.dll

- **目标**：消除跨 CRT 边界，根治堆腐败。
- **动作**：
  1. `build_model.bat:10-14` 移除 `vcvars64.bat`，改用 MinGW64 GCC 15.2（与 host 同 toolchain，同 ucrt/msvcrt）。
  2. 重编后 osdi_library.cpp:32-37 的"intentionally leak" workaround 可以拆除（回归测试验证）。
  3. 如果 BSIM4 源码（VerilogA via OpenVAF）对 GCC 编译有兼容问题（如 MSVC 专有 intrinsics、ABI 不同），需要小范围 patch OpenVAF 生成路径。
- **代价**：构建脚本改造 50 行 + OpenVAF 兼容性 unknown（中等风险）。
- **效果**：彻底解决 #2.2.1；#2.2.2 (NodeId=4) 仍需 GDB 验证。
- **回归保护**：保留 MSVC 编译路径作为 `build_model_msvc.bat`，CI 双轨直到 MinGW 路径 burn-in 通过。

#### 路径 B【不推荐】per-instance LoadLibrary

- **思路**：每个 OSDI 实例独立 LoadLibrary 一次 bsim4.dll，规避全局静态表共享。
- **否决理由**：
  1. Windows LoadLibrary 引用计数同名 dll 只 alloc 一份 DllMain，并不真正分离静态段；要真分离需 `LoadLibraryEx + DONT_RESOLVE_DLL_REFERENCES` 或重命名 dll 副本——后者每实例要 copy 文件，IO 灾难。
  2. 不解决 #2.2.1 跨 CRT 堆腐败本质。
  3. 实施复杂度 > 路径 A。

### 2.4 最小重现实验设计

#### 实验 E1：cross-CRT 是否真 root cause？

- 启用 Application Verifier 的 **PageHeap (Full)** 给 host exe；运行 `Bsim4ManyInstance_N15`。
- 期望：在 setup_model 第 ~14 次调用附近，AV 应抓到 `WAVES_HEAP_CORRUPTION` 或 `INVALID_FREE_ACCESS_VIOLATION`，调用栈一端在 bsim4.dll，一端在 host msvcrt。
- 若抓到 → 证实 2.2.1，立即推 **路径 A**。

#### 实验 E2：NodeId=4 是否硬 bug？

- 用 GDB / WinDbg 在 setup_instance 入口断点，dump BSIM4 dll 内 `_BSIM4_node_array[]` 地址（OpenVAF 生成的命名）；观察 N=3 第 4 个 NodeId 写入位置。
- 若发现写到 `[3]` 但数组定义 size=3 → OpenVAF/BSIM4 模板硬 bug；上游 issue。
- 若发现 array size 足够、index 也对 → 是 cross-CRT 堆腐败的次生症状（PageHeap 应能在 E1 抓到）。

- **A 完成后**：`Bsim4ManyInstance_N20` PASS（with MinGW-built dll）；osdi_library.cpp 内 intentional-leak 可移除并回归 100/100。

## 4. 风险与回归基线

### 4.1 必守红线

- **不允许**回归 `100/100 PASS` 基线（status0620_v5.md 表）。
- **不允许**回归 `RFSIM_DCOP_VERBOSE`、`autoHomotopy` 等已有环境变量行为。
- **不允许**修改 `osdi_library.cpp:32-37` 的 intentional-leak 直到路径 A 验证通过。

### 4.2 已知风险

1. **路径 B 的 oversampling** 把 FFT scratch 放大 4×，可能在 NH=8 时触发内存压力；需在 thread_local scratch 加 high-watermark log。
2. **路径 D 的 AC warm-start** 在某些 case（V_s=0 的零激励测试）变成 NaN；需要 guard `if (acMag<1e-15) skip warm-start`。
3. **路径 A 重编 bsim4.dll** 可能导致与既有 MSVC-built 模型 ABI 差异；需保留双 dll 副本期 ≥ 2 个 Sprint。

### 4.3 监控指标
